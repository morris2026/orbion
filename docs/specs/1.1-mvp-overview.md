## 1. MVP 定位与目标

MVP 的唯一目标：**跑通自我构建核心循环**，让 Orbion 可以用自己开发自己。

核心循环：

```
团队讨论 → 总结 Agent 产出摘要 → 分解 Agent 产出执行计划
→ 团队审批计划 → 执行 Agent 产出代码 → 团队审查 diff → Git commit → Orbion 自身增长
```

自我构建 9 点验证标准：

1. 能创建项目、添加成员、创建讨论线程
2. 人类能在讨论线程中发言
3. 总结 Agent 能订阅讨论事件，产出摘要
4. 分解 Agent 能订阅摘要事件，产出执行计划
5. 人类能审批执行计划（逐项 Approve/Reject）
6. 执行 Agent 能订阅审批事件，产出代码
7. 人类能审查代码 diff，Approve 或 Request Revision
8. 审批通过后能自动 Git commit
9. 前端三栏布局能实时展示讨论流和执行面板

## 2. 单服务架构

### 2.1 为什么单服务 MVP 合理

- 自我构建阶段只有一个项目（Orbion 自身），5 服务架构是过度工程
- 单服务部署简单（1 个进程 + 1 个 PostgreSQL），开发调试快
- 事件驱动架构在单服务内也工作 — 模块间通过 EventBus Protocol 通信
- 后期拆分为 5 服务只需把模块提取为独立进程，接口定义不变

### 2.2 模块划分

单服务内的模块分为两层：Hub 层（入站和基础设施）和 Business 层（业务逻辑）。Phase 2 拆分时 Hub 层保留为 Hub Gateway，Business 层模块提取为独立服务。

### 2.3 模块间通信

所有模块间通信通过 EventBus Protocol，不直接调用其他模块的 service 方法：

```
biz/threads/service.py 发布 DiscussionMessageCreated → EventBus
biz/agents/scheduler.py 订阅 DiscussionMessageCreated → 调度总结 Agent
biz/agents/runtime.py 调用 ModelAdapter → 发布 DiscussionSummaryGenerated → EventBus
biz/threads/service.py 更新 thread_messages 投影
hub/channels/sse.py (SSEChannel) 推送 SSE → 前端
```

唯一例外：`app/main.py` 在启动时初始化所有模块，注入 EventBus/EventStore 实例。这是组装逻辑，不是业务通信。

## 4. MVP 范围裁剪

### 4.1 Phase 1 — 做什么

| 子系统 | MVP 范围 | 说明 |
|--------|---------|------|
| 认证 | JWT + 用户名密码登录 + Admin审批注册 | register/login/审批，不做 OAuth；RegistrationPolicy可替换 |
| 项目 | 创建 + 列表 + 详情 | 只做 3 个端点 |
| 成员 | 添加成员 + 分配角色 | 创建者自动成为 Owner |
| 线程 | 创建 + 列表 | 状态只有 active/archived |
| 消息 | 发送 + 列表 + request_summary 标志 | 触发总结 Agent |
| 执行计划 | 列出 + 审批 + 拒绝 | 部分审批支持；MVP 裁剪 reviewing 中间状态（proposed 直接到 approved/rejected） |
| 产出 | 列出 + 审批 + 要求修改 | 审批后自动 Git commit |
| Agent | 3 个内置 Agent（总结/分解/执行） | 不做自定义 Agent |
| EventBus | InProcessEventBus | 进程内路由 |
| EventStore | PostgreSQL event_log | 不可变日志 |
| CQRS | 4 个 MVP 投影 | thread_messages, execution_plans, task_outputs, project_members |
| SSE | 1 个 SSEChannel 推送流 | 9 种项目级事件推送（不含平台级UserRegistered） |
| 权限 | 项目级 Bitmask RBAC | 不做空间级/资源级覆盖 |
| 记忆 | memory.md 层次化管理 | 平台/项目/Agent/任务 4 层 |
| Git | 审批后自动 commit | 不做 push，本地仓库 |
| 前端 | 三栏工作区 | 登录 → 工作区，无概览页 |
| Event Store 存储 | PostgreSQL 单表 + 投影表 | Phase 2 可迁移 Kafka + ClickHouse |
| 知识库实现 | Wiki 标签/链接导航 | Phase 3 可替换向量检索/GraphRAG |
| 计费 | 不实现 | Phase 3 subscription tracking |
| 手机 App | 不实现 | Phase 2 轻量版（查看/审批/通知） |

### 4.2 Phase 2 — 延后什么

| 子系统 | Phase 2 范围 |
|--------|-------------|
| 认证 | OAuth（GitHub/Google）、密码重置 |
| EventBus | NATSEventBus 替换 InProcessEventBus |
| 服务拆分 | 从单服务拆分为 5 服务 + NATS |
| 知识库 | Wiki CRUD + Context Injection |
| 权限 | 空间级覆盖 + 资源级覆盖 + ABAC |
| Agent | 自定义 Agent、Agent 间协作编排、Skill 市场 |
| 前端 | 知识空间页面、概览/统计 |
| 部署 | SaaS 多租户、计费层 |
| 手机 App | 轻量版（查看/审批/通知） |

## 5. 技术决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 服务架构 | 单服务 MVP | 自我构建只需 1 项目，5 服务过度工程 |
| 事件路由 | InProcessEventBus (asyncio) | MVP 单进程，零外部依赖，Phase 2 切换 NATS |
| EventBus 接口 | Protocol 抽象 | 切换实现时业务代码零改动 |
| 消息中间件 | 不引入（MVP） | PostgreSQL LISTEN/NOTIFY 不适合消费者组模式，InProcessEventBus 已满足 |
| Phase 2 中间件 | NATS JetStream | MIT 许可、极轻量、消费组+持久化，比 Valkey 更适合微服务 |
| 数据库 | PostgreSQL only | Event Store + 投影 + 业务数据一体化 |
| ORM | 不用 SQLAlchemy | MVP 用 asyncpg 直接写 SQL，更轻量更快 |
| Agent 模型 | Claude Haiku MVP | 月成本 ~$3.6，质量足够后再升级 Sonnet |
| 前端框架 | Vite SPA | 纯 SPA，构建产物为静态文件，由 Hub FastAPI 服务，无需额外 Node 进程 |
| 前端布局 | 三栏工作区 | 左栏导航、中栏讨论、右栏执行 |
| 知识库 | Phase 2 | MVP 不做，先跑通核心循环 |
| Python 环境 | uv | 比 pip/poetry 快 10-100 倍，自我构建阶段效率优先 |
| 前端构建 | Vite SPA | 纯静态产出，FastAPI StaticFiles 挂载，不需要独立 Node 运行时 |
| E2E 测试 | Playwright | 跨浏览器支持好，对 SSE 实时推送测试能力强 |
| CI/CD | MVP 不做 | 验证标准是自我构建 9 点验证，不是 CI green |
| 测试数据库 | 独立 orbion_test | 集成测试不污染开发数据，测试后自动清理 |
| Mock 策略 | 业务逻辑 mock，数据链路真实 DB | 权限计算不需要 DB，EventStore 必须真实 DB |
| Agent 测试 | mock ModelAdapter | 不消耗 API token，不依赖外部服务 |
| UserRepositoryProvider | ABC + 注册表 + scoped() | 与EventStore同模式self-managed pool；ABC而非Protocol因为connect/close/scoped是基础设施硬性方法 |
| 配置驱动实现选择 | orbion.json + 环境变量双层 | 非密钥配置可版本化追踪，密钥只从环境变量读取 |
| Provider.scoped() | 同步方法返回async context manager | sync仅构造实例无async操作，async工作在Repository.__aenter__；per-request事务隔离 |

## 6. 启动流程

`app/main.py` 的启动顺序：

```python
# 1. 加载配置（orbion.json + 环境变量）
settings = get_settings()

# 2. 初始化 EventBus（MVP 用 InProcessEventBus）
event_bus = InProcessEventBus()

# 3. 动态加载 EventStore 实现类并初始化（self-managed pool模式）
store_cls = load_store_impl(settings.event_store)
event_store = store_cls()
await event_store.connect()

# 4. 动态加载 EventProjections 实现类并初始化
projections_cls = load_projections_impl(settings.event_projections)
projections = projections_cls(event_bus)
await projections.connect()

# 5. 动态加载 UserRepositoryProvider 实现类并初始化（self-managed pool模式）
provider_cls = load_user_repo_provider(settings.user_repo)
user_repo_provider = provider_cls()
await user_repo_provider.connect()

# 6. 初始化 Agent Runtime
agent_runtime = AgentRuntime(event_bus, event_store, projections)

# 7. 注册内置 Agent（总结/分解/执行）
agent_runtime.register(summary_agent_declaration)
agent_runtime.register(decompose_agent_declaration)
agent_runtime.register(execute_agent_declaration)

# 8. 初始化 SSEChannel（ChannelPlugin 实现）
sse_channel = SSEChannel(event_bus)

# 9. 初始化 Git 集成
git_service = GitService(config.repo_path)

# 10. 组装 FastAPI app，注入依赖
app = FastAPI()
app.state.event_bus = event_bus
app.state.event_store = event_store
app.state.user_repo_provider = user_repo_provider
# ... 挂载各模块的 Router
```

所有模块通过 `app.state` 获取共享的 EventBus/EventStore/UserRepositoryProvider 实例，不通过全局变量或单例模式。路由通过 `provider.scoped()` 获取 per-request 事务 scoped 的 UserRepositoryProtocol 实例。

```bash
# 或应用启动时自动执行迁移

# 3. 启动后端（内嵌 Vite SPA 静态文件服务）
uvicorn app.main:app --reload --port 8000

# 4. 前端开发模式（独立 Vite 开发服务器，热更新）
cd web && npm run dev
# 前端开发服务器访问 http://localhost:5173，API 代理到 http://localhost:8000

# 5. 生产模式：构建前端 + 后端直接服务静态文件
cd web && npm run build    # 构建产出到 web/dist/
# uvicorn 自动挂载 web/dist/ 为静态文件，浏览器访问 http://localhost:8000
```

### 7.4 配置管理

所有配置分两层管理：orbion.json 非密钥配置 + ORBION_* 环境变量密钥覆盖。使用 Pydantic Settings：

```python
class Settings(BaseSettings):
    # 密钥字段：只从环境变量读取，不出现在 orbion.json 中
    jwt_secret: str = "orbion-dev-secret"  # 生产环境必须改
    anthropic_api_key: str = ""

    # 非密钥字段：orbion.json 基础值，环境变量可覆盖
    event_store: str = "postgres"
    event_projections: str = "postgres"
    user_repo: str = "postgres"
    postgres: PostgresSettings = PostgresSettings()
    repo_path: str = "./repo"  # Git 产出存放路径
    memory_base_path: str = "./data/memory"

    model_config = SettingsConfigDict(env_prefix="ORBION_", env_nested_delimiter="__")
```

PostgresSettings 是嵌套子配置，含 host/port/db/user/password + computed url（`postgresql://user:password@host:port/db`）。密钥字段（jwt_secret、anthropic_api_key、postgres.password）只从 ORBION_* 环境变量读取，不允许出现在配置文件中。

开发时使用默认值，生产时通过环境变量覆盖：`ORBION_JWT_SECRET=...`、`ORBION_ANTHROPIC_API_KEY=...`、`ORBION_POSTGRES__PASSWORD=...`。

## 9. 测试策略

### 9.1 后端测试

| 类型 | 工具 | 覆盖范围 | 数据库 |
|------|------|---------|--------|
| 单元测试 | pytest + pytest-asyncio | Pydantic 模型、权限计算、事件 payload 解析 | 不需要 |
| 集成测试 | pytest + asyncpg | EventStore 写入/查询、投影更新、完整 API 端点 | PostgreSQL（docker） |
| API 测试 | httpx + FastAPI TestClient | 请求/响应校验、认证流程、权限检查 | PostgreSQL（docker） |

### 9.2 集成测试数据库隔离

每个集成测试用独立的测试数据库，避免污染开发数据。基础设施（EventStore、UserRepositoryProvider）使用 self-managed pool 模式初始化：

```python
# tests/integration/conftest.py
# 基础设施初始化使用Provider模式（与main.py一致）
settings = get_settings()
store_cls = load_store_impl(settings.event_store)
provider_cls = load_user_repo_provider(settings.user_repo)

app.state.event_store = store_cls()
await app.state.event_store.connect()
app.state.user_repo_provider = provider_cls()
await app.state.user_repo_provider.connect()

# 测试基础设施可使用asyncpg直接验证DB状态（测试本身不属于业务代码）
db_conn = await asyncpg.connect(settings.postgres.url)
```

### 9.3 前端测试

| 类型 | 工具 | 覆盖范围 |
|------|------|---------|
| 单元测试 | Vitest | 组件逻辑、状态管理、API 调用封装 |
| E2E 测试 | Playwright | 登录流程、消息发送、审批操作、SSE 实时更新 |

Playwright 覆盖的自我构建核心循环：

1. 注册（首个用户自动审批）→ 登录 → 创建项目
2. 创建线程 → 发送消息 → request_summary
3. 验证总结 Agent 产出摘要出现在消息流
4. 审批执行计划（approve 部分 tasks）
5. 验证执行 Agent 产出 diff 出现在右栏
6. 审批产出 → 验证 Git commit
7. SSE 实时推送验证（消息出现时前端自动更新）

### 9.5 测试运行

```bash
# 后端单元测试（无需数据库）
pytest tests/unit/ -v

# 后端集成测试（需要 PostgreSQL）
docker compose up -d
pytest tests/integration/ -v

# 前端单元测试
cd web && npm run test

# 前端 E2E 测试（需要完整环境：PostgreSQL + 后端）
# 后端直接服务前端静态文件，无需额外前端进程
cd web && npm run test:e2e
```

### 9.6 测试与真实数据库 vs Mock

原则：**业务逻辑测试 mock，数据链路测试真实数据库**。

- Pydantic 模型校验、权限计算、事件 payload → 纯 Python，无需数据库
- EventStore 写入/查询、投影更新、API 端点 → 必须用真实 PostgreSQL
- Agent 调用 LLM → mock ModelAdapter（不消耗 API token，不依赖外部服务）

## 10. 部署

### 10.1 MVP 部署方式

私有化部署，单机运行，2 个进程：

```
进程 1: PostgreSQL (docker)
进程 2: uvicorn (Python 后端 + Vite SPA 静态文件服务)
```

浏览器直接访问 uvicorn 进程（端口 8000），API 和前端页面由同一进程服务。

### 10.2 docker-compose.yml

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: orbion
      POSTGRES_USER: ${ORBION_PG_USER}
      POSTGRES_PASSWORD: ${ORBION_PG_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    restart: always

  backend:
    build: .
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000
    environment:
      ORBION_POSTGRES_URL: postgresql://${ORBION_PG_USER}:${ORBION_PG_PASSWORD}@postgres:5432/orbion
      ORBION_JWT_SECRET: ${ORBION_JWT_SECRET}
      ORBION_ANTHROPIC_API_KEY: ${ORBION_ANTHROPIC_API_KEY}
    ports:
      - "8000:8000"
    depends_on:
      - postgres
    restart: always

volumes:
  pgdata:
```

后端 Dockerfile 构建步骤包含前端：

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY app/ app/
COPY migrations/ migrations/
# 构建前端（需要 Node.js）
RUN apt-get update && apt-get install -y nodejs npm && rm -rf /var/lib/apt/lists/*
COPY web/ web/
RUN cd web && npm install && npm run build
# 最终只需要 web/dist/ 静态文件，不需要 Node 运行时
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 11. CI/CD

### 11.1 MVP 策略

自我构建阶段不需要正式 CI/CD。验证标准是 **自我构建 9 点验证**（见第 1 节），而非 CI green。

每次重大变更的验证流程：

```bash
# 手动验证脚本
docker compose up -d                          # 启动环境
pytest tests/unit/ -v                          # 单元测试
pytest tests/integration/ -v                   # 集成测试
cd web && npm run test:e2e                     # E2E 测试
# 人工走一遍核心循环：讨论 → 总结 → 分解 → 审批 → 执行 → commit
```

### 11.2 Phase 2 CI

自我构建循环跑通后，自然引入 CI：

- GitHub Actions：push 时自动跑单元 + 集成测试
- PR 合入前必须 CI green
- E2E 测试 nightly run（不是每次 push，太慢）
