# 多用户并行与 Git Worktree 模型

> 配套文档：
>
> - [Agent Runtime 重构设计](../superpowers/specs/agent-runtime-refactor-design.md)
> - [产出物管理与变更传播](../superpowers/specs/artifact-management-design.md)
>
> 本文档专门定义多用户/多 Agent 在同一项目并行操作 git 仓库的隔离与合并机制。

## 1. 问题背景

Orbion 是"多人 + 多 Agent 双协作"平台，必须正面解决：

| 层面         | 问题                                                     |
| ------------ | -------------------------------------------------------- |
| 工作目录隔离 | A 改文件时 B 看到的是哪个版本？Agent 执行基于哪个版本？  |
| 写入串行化   | 两个 Agent 同时改同一文件？两个用户同时编辑同一文件？    |
| Commit 合并  | 多个任务产出都审批通过，如何合并到主分支？冲突如何处理？ |

**核心原则**：半成品不可见（worktree 之间彻底隔离，避免污染和不可靠的中间状态可见性）。

## 2. 工作目录布局

```
projects/{project_id}/
  repo/
    {repo_name}.git/              ← bare 仓库（中心，不直接编辑）
    worktrees/
      main/                       ← 主分支 worktree（默认浏览入口，task 合并目标）
      task_{task_id}/             ← Agent 任务的隔离 worktree
```

每个 worktree 是独立的工作目录，共享同一 `.git` 仓库（`git worktree` 原生支持）。

**Orbion 不支持手动开发**——用户手动开发用本地 VSCode 等工具。Orbion 只负责 AI 协作开发流程，worktree 只有 main 和 task 两种。

## 3. Worktree 分类

| 类型     | 创建时机                  | 用途                        | 谁可访问                                   |
| -------- | ------------------------- | --------------------------- | ------------------------------------------ |
| **main** | 项目初始化                | 默认浏览入口，task 合并目标 | 所有项目成员**只读**；写入仅由合并流程完成 |
| **task** | Agent dispatch 时自动创建 | Agent 任务执行隔离          | owner（task 执行者）可读可写，其他人只读   |

**main 只读原则**：main 是合并目标，不允许人工直接编辑。任何修改（代码或设计文档）必须走 task worktree 流程，审批通过后由合并流程写入 main。这是"半成品不可见"原则的延伸——main 上看到的永远是已审批的稳定状态。

## 4. File tab 增强：worktree 选择能力

file tab 顶部新增 worktree 选择器，纯 git 视角，不混入业务审批：

```
┌────────────────────────────────────────────┐
│ [worktree: main ▼]                        │
├────────────────────────────────────────────┤
│ 文件树（基于选中 worktree）                │
│ ...                                        │
├────────────────────────────────────────────┤
│ 文件编辑器（编辑选中 worktree 的文件）     │
│ ...                                        │
├────────────────────────────────────────────┤
│ Source Control（针对选中 worktree）        │
│ - staged / unstaged / commits              │
└────────────────────────────────────────────┘
```

下拉选项：

```
[worktree: main ▼]
   ├─ main
   ├─ task_001 (Agent 执行中)
   ├─ task_002 (待审批)
   └─ task_003 (审查中)
```

操作：

- 切换 worktree → 文件树/编辑器/Source Control 全部重新加载对应版本
- **没有"新建 worktree"按钮**——用户不能主动创建 worktree

**职责边界**：

- file tab 的 Source Control 只展示 git 视角的变更（staged/unstaged/commits）
- 业务级的产出审批（待审批 Agent 产出 + diff 预览 + 批准/要求修改按钮）放在流程 tab 实施阶段
- 不在 file tab 内做"按 Agent 产出分组"或"审批按钮"，保持纯 git 视角

## 5. 并发修改解决方案

### 5.1 不同 worktree：完全隔离

不同 worktree 的修改互不可见，符合"半成品不可见"原则。

- A 在 `main` 改文件，B 在 `task_001` 看不到
- 合并时通过 git merge 处理

### 5.2 同一 worktree 同一文件：mtime 检测 + 自动三方合并

多用户/多 Agent 同时编辑同一 worktree 的同一文件时，采用**mtime 检测 + 自动三方合并 + 仅真冲突时弹窗**的策略：

```
B 打开文件
  ↓ 前端记录 original content + mtime
B 编辑文件
  ↓ 编辑器内容变化
B 保存
  ↓ 后端检测当前文件 mtime vs 前端传入的 expected_mtime
  ├─ 相同 → 直接保存（无冲突）
  └─ 不同 → 三方合并
       ├─ git merge-file 成功 → 直接保存合并结果（B 无感）
       └─ git merge-file 有冲突 → 返回 409，前端弹冲突对话框
```

#### 5.2.1 三方合并实现

利用 `git merge-file` 命令：

```bash
git merge-file --diff3 \
  /tmp/mine.py \      # B 的修改（前端传入）
  /tmp/original.py \  # 共同祖先（前端传入，B 打开时的内容）
  /tmp/theirs.py      # 磁盘当前内容（A 的版本）
```

- 返回 0：无冲突，`/tmp/mine.py` 变为合并结果
- 返回非 0：有冲突，`/tmp/mine.py` 包含冲突标记

#### 5.2.2 关键设计

- **original 由前端记录**：B 打开文件时的内容作为三方合并的"共同祖先"，减少后端状态管理
- **合并成功不通知 B**：操作成功是预期行为，无需额外提示
- **真冲突才弹窗**：只有同位置修改导致 git merge-file 失败时，才弹冲突对话框

**与 git 原生 merge 的语义偏差**：前端记录的"共同祖先"是 B 打开时的版本，非 git 历史的真实 merge-base。若 B 打开后长时间不保存（期间 main 已合并多次提交），三方合并的结果可能偏离 `git merge` 的语义。**适用边界**：打开超过 30 分钟未保存时，前端提示"文件已过期，建议重新加载"，避免基于陈旧 original 合并。MVP 接受此偏差（简化后端状态管理）；SaaS 阶段可改为后端查 git merge-base。

#### 5.2.3 冲突处理流程

冲突发生时的处理遵循 Orbion "AI 自动执行 + 人工确认"核心模式：

```
后端检测到冲突 → 409 响应（含 merged_content + conflict_markers）
  ↓
前端自动 dispatch Agent 尝试解决冲突
  - Agent 拿到 merged_content（含冲突标记）+ 文件上下文
  - Agent 产出解决后的版本
  ↓
┌─────────────────┴─────────────────┐
│ Agent 解决成功                     │ Agent 无法解决
│                                   │
↓                                   ↓
展示解决后的版本给用户确认          提示"冲突无法自动处理，需人工介入"
用户确认 → 保存                     展示冲突标记供手动编辑
用户拒绝 → 回到编辑器               用户手动解决后保存
```

**信息透明原则**：前端始终告知用户当前状态——"冲突已自动处理，待确认" / "冲突需人工介入"。用户无需主动发起解决，只需确认或介入。

**MVP fallback**（Agent Runtime 未集成时）：退化为 3 选项对话框——

| 选项       | 行为                                                      |
| ---------- | -------------------------------------------------------- |
| 对比并合并 | 编辑器载入含冲突标记的合并结果，用户手动解决后重存        |
| 覆盖       | 用当前用户的版本覆盖磁盘（对方的修改丢失）                |
| 取消       | 不保存，用户的编辑仍在编辑器                              |

Agent Runtime 集成后，"对比并合并"选项由 Agent 自动执行替代——Agent 尝试解决，用户只做最终确认。

#### 5.2.4 实现细节

- 文件保存 API 接收 `expected_mtime` + `original_content` + `current_content` 参数
- 后端检测 mtime 变化，变化时执行三方合并
- 合并成功：原子写入合并结果（tmp + rename）
- 合并失败：返回 409 Conflict，附带冲突内容
- 前端用 Monaco Diff Editor 展示冲突

### 5.3 同一 worktree 不同文件：无锁

不同文件的编辑互不阻塞，用户感知不到锁。

### 5.4 Commit 操作：全局 asyncio.Lock 串行化

git 操作（add/commit/merge/worktree add/remove）不能并发，否则会损坏共享的 `.git` 目录（refs、objects、packed-refs）。所有可能修改 `.git` 内部状态的操作通过全局 `asyncio.Lock` 串行化：

```python
# 进程级全局锁（按 event loop 缓存，兼容 pytest-asyncio function-scope loop）
_global_git_lock = asyncio.Lock()

async def commit_to_worktree(worktree_id, ...):
    async with _global_git_lock:
        # 执行 git add + commit
```

#### 5.4.0 git 命令调度与超时

- **async 调度**：`GitCommandService` 的方法是同步 `subprocess.run`，WorktreeService 在锁内通过 `await asyncio.to_thread(self._git.<method>, ...)` 调度，避免 subprocess 阻塞 event loop。锁本身是 async 的——若直接同步调用，subprocess 期间其他协程无法调度，锁退化为"自然串行"且整个服务冻结。
- **subprocess 超时**：`GitCommandService._run` 设 30s 超时（`_CMD_TIMEOUT`），超时返回 `returncode=124` 的失败 `CompletedProcess`，由上层翻译为 `WorktreeCreateError` / `WorktreeArchiveError`。Why：git 命令可能因 ssh 挂起、gc 慢等原因无限阻塞，在全局锁内持锁挂起会冻结整个进程。
- **DB 与 git 原子性**：`create_or_reuse` 的 git worktree add 与 DB INSERT 非天然原子；DB 失败时 try/except 回滚 git 操作（worktree remove + branch delete），避免孤儿 worktree。回滚失败记 warning 不抛错（孤儿残留由运维清理）。


#### 5.4.1 为什么需要全局锁

1. **git 自己的文件锁竞争**：git 操作时创建 `.git/index.lock`，并发会报错
2. **所有 worktree 共享 .git 目录**：refs、objects、packed-refs 是共享的
3. **merge 操作的原子性**：merge 涉及多步骤，并发会产生中间状态

#### 5.4.2 为什么不做按 worktree 分锁

- 不同 worktree 的某些操作（如 add）理论上可并行
- 但 refs、objects 等共享文件仍需全局锁
- 按 worktree 分锁复杂度高，收益有限
- git 操作通常毫秒级，串行可接受

**跨项目阻塞说明**：全局 `_global_git_lock` 跨所有项目共享，单实例下高并发项目会互相阻塞。MVP 可接受（单实例并发量低）；若实际运行中出现锁竞争（`orbion_git_lock_wait_seconds` metric 持续 > 1s），可改为按 `repo_name` 分锁（每个 bare 仓库一把锁），不同项目不互相阻塞。

#### 5.4.3 多实例部署

`asyncio.Lock` 只在单进程内有效。多实例部署时需改用 PostgreSQL advisory lock 等跨进程锁机制。**MVP 不考虑多实例**，advisory lock 推迟到 SaaS 化或大规模部署时再做。届时归档任务的跨进程锁策略一同升级（详见 [Agent Runtime 13.2 节](../superpowers/specs/agent-runtime-refactor-design.md#132-归档流程)）。

## 6. Worktree 生命周期

### 6.1 task worktree：与 task 生命周期对齐

**每 task 一个 worktree**：所有 task type（analysis / design / development）都有 worktree——所有产出物都跟仓库走。task 内多次 dispatch（重试/修改/恢复）复用同一 worktree，保留中间状态。task 进入终态（completed / cancelled）后保留 worktree 供审批，artifact 审批结束后清理。

```
[task 创建] ──→ [pending: 无 worktree]
                    ↓ dispatch
                [running: worktree active]
                    ↓
              ┌─────┴─────┐
        [用户 stop]     [Agent 完成]
              │              │
        [paused:            [completed: worktree 保留，artifact → proposed]
         worktree 保留]         ↓
              │           ┌─────┴─────┐
        [用户 resume  →   [artifact     [artifact rejected:
         running]          approved:     task 回到 pending]
                           worktree       用户可重新 dispatch]
                           保留]
              │                │
              ↓                ↓
        [worktree        [worktree
         archived]       archived]

        [timeout: worktree 保留，用户可重新 dispatch → running]
        [cancelled(cancel): worktree archive]
```

| task 状态   | worktree 状态 | 说明                                     |
| ----------- | ------------- | ---------------------------------------- |
| pending     | 不存在 / 保留 | 首次创建无 worktree；失败/崩溃/artifact rejected 后回 pending 保留旧 worktree |
| running     | active        | dispatch 时创建/复用，Agent 在内执行      |
| paused      | 保留          | 用户暂停，worktree 保留半成品供 resume    |
| completed   | active        | Agent 产出完成，artifact 进入 proposed，worktree 保留 |
| timeout     | 保留          | 超时后 worktree 保留，用户可重新 dispatch |
| cancelled   | archived      | cancel 后 archive worktree（终态清理）   |

**task 状态机统一**：本文档定义 task 状态的权威取值与流转，[产出物管理](../superpowers/specs/artifact-management-design.md#1114-变更传播机制)和 [Agent Runtime](../superpowers/specs/agent-runtime-refactor-design.md#34-任务数据模型tasks-表) 中的 task status 引用此处。task 有 6 个状态（pending / running / paused / completed / timeout / cancelled），审批流程在 artifact 层面（proposed → approved / rejected），不在 task 层面。

### 6.2 task 截止日期 + 提醒

- task 创建时可设置 `due_date`（可选）
- 超过 `due_date` 仍未 completed → 给用户提醒（**不改 task status**）
- 提醒方式：流程 tab 显示过期标识、讨论区 @用户提示、通知中心
- 用户决定后续：继续推进 / 重新分配 / cancel
- 不阻塞 task 流转，用户仍可继续 dispatch

### 6.3 worktree 状态汇总

| 状态          | 含义                                       |
| ------------- | ------------------------------------------ |
| `active`      | 活跃中，可读可写                           |
| `conflicting` | 合并时冲突，worktree 保留（等待重生成）    |
| `archived`    | 已归档/删除                                |

### 6.4 WorktreeService 接口签名

`WorktreeService` 是 worktree 生命周期的权威管理接口，[agent-runtime 8.3](../superpowers/specs/agent-runtime-refactor-design.md#83-worktreeservice-协作) 引用本接口。

**依赖注入**：`create_or_reuse(task_id)` 单参数签名，但创建 worktree 需要 `project_id` / `repo_name` / `owner_user_id`（定位 bare 仓库 + 写 worktrees 表 created_by）。这些信息存于 tasks 表（[agent-runtime §3.4](../superpowers/specs/agent-runtime-refactor-design.md#34-任务数据模型tasks-表)），通过 `TaskResolver` 协议注入，解耦 WorktreeService 与 tasks 表的直接依赖：

```python
@dataclass
class TaskContext:
    project_id: UUID
    repo_name: str
    owner_user_id: UUID
    task_status: str | None  # task 当前状态，delete_by_owner 校验是否允许放弃（None 表示未知）

class TaskResolver(Protocol):
    async def resolve(self, task_id: UUID) -> TaskContext: ...
    # task 不存在时抛 KeyError，由 WorktreeService 翻译为 TaskNotFoundError
```

MVP 阶段用内存 stub；agent-runtime 重构后实现 `PostgresTaskResolver` 查 tasks 表。

```python
class WorktreeService:
    def __init__(
        self,
        git: GitCommandService,
        settings: Settings,
        pool: asyncpg.Pool,
        task_resolver: TaskResolver,
    ) -> None: ...

    # 只读 property：pool / task_resolver 暴露供测试与上层诊断访问
    # （测试需查 DB 表状态、注册 task context；上层需注入 TaskResolver 实例）
    @property
    def pool(self) -> asyncpg.Pool: ...
    @property
    def task_resolver(self) -> TaskResolver: ...

    async def create_or_reuse(self, task_id: UUID) -> Worktree:
        """为 task 创建或复用 worktree。
        - 入口校验 repo_name 路径安全（^[a-zA-Z0-9_.\\-]+$，防 TaskResolver 外部输入路径遍历）
        - 首次 dispatch：git worktree add worktrees/task_{task_id} -b task/{task_id} main
          - DB INSERT 失败时回滚 git 操作避免孤儿 worktree（5.4.0）
        - revision 重做：复用现有 worktree，保留之前产出
        返回 Worktree（含 path / branch_name / status）。
        异常：
          - TaskNotFoundError（task_id 在 TaskResolver 中未注册）
          - WorktreeCreateError（git worktree add 失败 / DB INSERT 失败含回滚后仍失败）
          - GitOperationError（git fetch / branch 创建失败）
        """

    async def merge(self, task_id: UUID, target_branch: str = "main") -> MergeResult:
        """合并 task worktree 到目标分支。
        - 加全局 git 锁（5.4）
        - git merge task/{task_id} 到 target_branch
        - 冲突时调 regenerate() 自动重生成（8.1）
        返回 MergeResult（success / conflict / error）。
        异常：
          - WorktreeConflictError（冲突且重生成次数超限，详见 8.1）
          - GitOperationError（merge 内部异常）
          - WorktreeNotFoundError（task 无 active worktree，无法 merge）
        """

    async def regenerate(self, task_id: UUID) -> Worktree:
        """冲突重生成：删除当前 worktree + 分支，基于最新 main 重建。
        - conflict_regen_count 递增（详见 8.1）
        - 超过 MAX_CONFLICT_REGENERATIONS 抛 WorktreeConflictError
        异常：
          - WorktreeConflictError（重生成次数超限）
          - WorktreeNotFoundError（task 无 active worktree）
        """

    async def archive(self, task_id: UUID) -> None:
        """归档 task worktree（合并/拒绝/放弃后调用）。
        - 入口校验 repo_name 路径安全（^[a-zA-Z0-9_.\\-]+$）
        - worktree 不存在（pending 首次）：no-op，不抛错
        - worktree 存在：加全局锁 → git worktree remove + branch delete + status='archived'
          - git branch -D task/{task_id}（archive 场景产出已合并或显式丢弃，显式传 force=True）
          - 锁内 double-check 防并发 archive 同一 task
        异常：
          - WorktreeArchiveError（git worktree remove 或 branch delete 失败）
        """

    async def delete_by_owner(self, worktree_id: UUID, user_id: UUID) -> None:
        """task 执行者主动放弃（10.3 入口）。这是 AgentRuntime.cancel(task_id) 在 worktree 层的入口——
        AgentRuntime.cancel 负责 task 状态流转（→cancelled）+ artifact 处理（→rejected），
        内部调用本方法完成 worktree archive。本方法只负责 worktree 层操作。
        - 校验 user_id 是 worktree owner
        - 校验 task 状态为 pending/running/paused/timeout（completed/cancelled 不允许放弃）
        - 调用 archive()；pending 首次无 worktree 时 archive 为 no-op（无文件可删，不抛错，仅标记 task cancelled）
        异常：
          - PermissionError（非 owner）
          - TaskStateError（task 状态不允许放弃）
        """

    async def get(self, worktree_id: UUID) -> Worktree | None: ...
    async def list_by_project(self, project_id: UUID) -> list[Worktree]: ...
```

**与 agent-runtime 的契约**：`AgentRuntime.dispatch` 调用 `create_or_reuse()`；`AgentRuntime.cancel(task_id)` 内部调用 `delete_by_owner()`（进而调 `archive()`）完成 worktree 清理；审批通过后调用 `merge()`；崩溃恢复时不直接调用 WorktreeService（由 task 状态驱动，reconcile 将 task 回到 pending，用户重新 dispatch 时 `create_or_reuse` 复用保留的 worktree）。

### 6.5 资源容量估算

| 资源 | 估算 | 说明 |
| ---- | ---- | ---- |
| 单 task worktree 磁盘占用 | 50-500 MB | 取决于项目代码量 + Agent 产出；git worktree 共享 .git/objects，仅工作目录独立 |
| 单项目并发 task worktree 数 | 默认上限 20（`runtime.json: max_worktrees_per_project`） | 防止磁盘爆炸；超出时新 dispatch 排队 |
| 单用户并发 task worktree 数 | 默认上限 5 | 同 1.1 并发上限 |
| bare 仓库大小 | 项目代码量的 1-2 倍 | git 压缩后通常小于工作目录 |
| 监控指标 | 磁盘使用率 > 80% 告警 | P2 告警，运维清理 archived worktree |

**清理策略**：`archived` 状态的 worktree 目录立即删除（git worktree remove），仅保留 worktrees 表记录用于审计。`archived` 记录保留 90 天后批量 DELETE（详见 [agent-runtime 15.5](../superpowers/specs/agent-runtime-refactor-design.md#155-数据保留策略)）。

## 7. Agent 任务的 Worktree 流程

```
1. 用户触发 /implement <task> 或 @implementer <task>
   ↓
2. dispatch 时：
   - 检查 task 是否已有 worktree
     ├─ 无（首次 dispatch）：
     │   - git fetch（同步远端）
     │   - git worktree add worktrees/task_{task_id} -b task/{task_id} main
     │   - 任务 worktree 基于主分支创建
     └─ 有（revision 重做）：
         - 复用现有 worktree（保留之前产出）
   ↓
3. Agent 在 task worktree 中执行：
   - 读文件：从 worktrees/task_{task_id}/ 读
   - 写文件：写到该目录
   - 调用 Skill：在 worktree 内执行
   - revision 场景：Agent 能看到之前产出 + 修改意见，基于此改进
   ↓
4. Agent 产出 diff（与 task worktree 的 HEAD 比较）
   注：worktree 创建在 dispatch 流程的步骤 2-3 之间，详见 [Agent Runtime 第 8 节 dispatch 流程](../superpowers/specs/agent-runtime-refactor-design.md#8-agent-runtime-调度改造)
   ↓
5. 用户在 file tab 审查 diff（可切换到 task worktree 查看）
   ↓
6. 审批通过 → 合并流程（第 8 节）→ 清理 worktree
   审批拒绝 → task 回到 pending，worktree 保留（半成品不丢失），用户可修改后重新 dispatch 或放弃
```

## 8. 合并流程

任务审批通过后的合并：

```python
async def merge_task_to_target(task_id, target_branch="main"):
    async with global_git_lock:
        # 1. 切换到目标 worktree
        target_wt = worktrees[target_branch]
        # 2. fetch 远端最新（如果有远端）
        # 3. 尝试 merge
        result = git.merge(f"task/{task_id}", cwd=target_wt)
        if result.has_conflicts:
            # 中止合并，回到合并前状态
            git.merge("--abort", cwd=target_wt)
            # 自动触发重新生成（基于最新 main）
            await regenerate_task(task_id)
            return
        # 4. 清理 task worktree（MVP 不做远端推送，详见第 10 节）
        git.worktree.remove(f"worktrees/task_{task_id}")
        git.branch("-D", f"task/{task_id}")
```

### 8.1 冲突处理：自动重新生成

冲突时不提供"手动解决"或"让 Agent 解决冲突"选项，统一走**自动重新生成**：

```
合并冲突
  ↓
task 保持 completed，worktree 标记 conflicting（短暂，用于通知用户）
  ↓
中止合并，main 回到合并前状态
  ↓
删除当前 task worktree（含冲突的产出）
  ↓
基于最新 main 创建新 worktree
  ↓
重新 dispatch Agent（与首次 dispatch 流程一致）
  ↓
Agent 产出 → 走正常审批流程（含代码评审）
```

**重生成次数上限**：单个 task 累计重新生成不超过 `MAX_CONFLICT_REGENERATIONS`（默认 3，项目级可配 `runtime.json` 中 `max_conflict_regenerations`）。超过后 task 回到 `pending`，在流程 tab 提示"冲突次数过多，需人工介入"，由用户决定：

- 手动调整 task 后重新 dispatch（重置计数）
- 取消 task（task.status → cancelled）

**计数器独立**：`conflict_regen_count` 存于 `tasks.conflict_regen_count` 字段，与 critic 的 `revision_count`（`tasks.revision_count`，[agent-runtime 7.5](../superpowers/specs/agent-runtime-refactor-design.md#75-递归防护与冲突处理)）独立计数，互不影响。前者计冲突重生成次数，后者计 critic 拒绝导致的重做次数。

**重做场景的修改意见继承**：重新生成时若 task 之前有审批者/critic 的修改意见，新 dispatch 的 Agent prompt 必须携带原修改意见列表，避免丢失上下文。`ImplementationRequested` 事件的 payload 含 `revision_notes` 字段（详见 [Agent Runtime 第 3 节](../superpowers/specs/agent-runtime-refactor-design.md#3-事件-payload-定义)）。

**用户控制**：

- 默认自动重新生成，用户在流程 tab 看到"冲突，已自动重新生成（第 N 次/共 3 次）"通知
- 用户可在冲突时取消（task.status → cancelled）
- 未超次数前每次都基于最新 main 自动重生成

**关键设计**：

- 重新生成后的产出走**正常代码评审流程**，与普通产出一致
- 不为冲突场景做特殊审批处理
- task 历史保留"曾发生冲突并重新生成"的记录，便于追溯

## 10. 安全与权限

**统一原则**：main worktree 对所有用户只读（写入仅由合并流程完成）；task worktree 的 owner 可写，其他项目成员只读。不做 worktree 独立权限配置，沿用项目级权限继承。

### 10.1 各类 worktree 的权限

| worktree 类型 | owner                           | owner 权限                  | 其他人权限 |
| ------------- | ------------------------------- | --------------------------- | ---------- |
| main          | 项目本身                        | **只读**（合并流程除外）    | 只读       |
| task          | task 执行者（触发 task 的用户） | 可读可写（含 Agent 代执行） | 只读       |

main worktree 不开放任何写操作 API。代码产出和设计文档的修改都必须走 task worktree 流程，审批通过后由合并流程写入 main。

### 10.2 创建权限

- **main worktree**：项目初始化时自动创建
- **task worktree**：系统在 Agent dispatch 时自动创建（owner 是触发 task 的用户）
- **用户不能主动创建 worktree**（Orbion 不支持手动开发）

### 10.3 删除权限

- main：不可删
- task worktree：task 完成（合并/拒绝）后系统自动清理；task 执行者可主动放弃

  **task 执行者放弃 task 的入口**：
  - UI：流程 tab 中该 task 卡片的"放弃"按钮
  - API：`DELETE /projects/{project_id}/worktrees/{worktree_id}`（详见 13.1）
  - 调用后系统删除 task worktree + 分支，task 状态置为 `cancelled`，记录"owner 主动放弃"原因

### 10.4 Agent 代执行

Agent 在 task worktree 中操作时，以 task 执行者身份执行：

- Agent 的文件读写、git 操作都记录在 task 执行者名下
- commit author 是 task 执行者（详见第 9 节）
- 这意味着 task 执行者对 task worktree 有"代执行"权，Agent 是其代理人

### 10.5 跨用户协作审查

其他用户可以切换到 task worktree 只读查看（用于审查 Agent 产出）：

- 切换到 task worktree → 文件浏览/编辑器以只读模式加载
- 不能编辑、不能 commit、不能 push
- 可以评论、可以发起审查请求

## 11. 数据模型

新增 `worktrees` 表（PostgreSQL）：

```
worktrees 表
├── id              UUID PK
├── project_id      UUID FK
├── repo_name       VARCHAR
├── worktree_type   VARCHAR(16)   ← main | task
├── branch_name     VARCHAR
├── path            VARCHAR       ← worktree 在文件系统的绝对路径
├── status          VARCHAR(16)   ← active | conflicting | archived
├── created_by      UUID          ← 创建者 user_id（task 类型为系统）
├── task_id         UUID NULL     ← task 类型关联的任务 ID
├── created_at      TIMESTAMP
└── updated_at      TIMESTAMP
索引：
  INDEX(project_id, worktree_type)
  INDEX(task_id) WHERE task_id IS NOT NULL
  UNIQUE(project_id, branch_name) WHERE status != 'archived'
  UNIQUE(task_id)              WHERE task_id IS NOT NULL AND status != 'archived'
```

**唯一约束说明**：活跃 worktree 在 DB 层兜底防并发竞态——`create_or_reuse(task_id)` 在应用层查重时，并发 dispatch 同一 task 会产生竞态，两条 `worktree_type='task' task_id=<same>` 记录可能同时插入。部分唯一索引（`WHERE status != 'archived'`）保证同项目同分支、同 task 同时只能有一个活跃 worktree；archived 记录保留多条用于审计。`task_id` 索引和唯一约束均为部分索引，main worktree 的 NULL 行不进入索引。

## 15. 远端仓库同步

**MVP 阶段拉取支持、推送暂缓**：远端→本地的同步（clone、fetch）MVP 支持；本地→远端的推送（push）暂缓，因为推送策略（推哪个分支、冲突如何处理、是否经审批）尚未定。本地 bare 仓库 + worktree 模型不依赖推送即可工作。

| 时机                 | 操作                              | MVP 状态 |
| -------------------- | --------------------------------- | -------- |
| 项目初始化           | `git clone --bare {remote_url}`   | 支持（拉取） |
| 任务 worktree 创建前 | `git fetch origin` 同步远端最新   | 支持（拉取） |
| 合并到 main 后       | `git push origin main` 推送到远端 | 暂缓（推送） |
| 用户分支 push        | 显式操作（按钮或命令）            | 暂缓（推送） |

推送冲突处理（暂缓）：

- push 失败（远端有新提交）→ fetch + merge/rebase → 重试 push
- 多次失败 → 提示用户手动处理

### 16.2 集成测试

- **worktree 生命周期**：创建 task → dispatch → 产出 → 审批通过 → 合并 → archive，验证 worktrees 表状态流转与文件系统一致
- **冲突重生成端到端**：构造 main 推进 → task 合并冲突 → 自动重生成（最多 3 次）→ 超限 task 回到 pending
- **三方合并端到端**：两用户同时编辑同文件 → 后保存者触发三方合并 → 自动合并成功 / 真冲突弹窗
- **崩溃恢复**：dispatch 中途模拟进程重启 → worktree 保留 → owner 重新 dispatch 复用 worktree
- **跨项目隔离**：两项目并发 dispatch → worktree 路径不冲突 → git 锁串行化但不死锁

### 16.3 覆盖率门槛

- `WorktreeService` 接口：行覆盖 ≥ 95%（生命周期关键）
- 三方合并逻辑：行覆盖 ≥ 90%
- 全局锁：分支覆盖 ≥ 85%（并发场景关键）
