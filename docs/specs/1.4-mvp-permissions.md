## 1. Bitmask 权限位定义

### 1.1 人类权限位

| Bit | 权限名 | 说明 |
|-----|--------|------|
| 0 | VIEW_DISCUSSION | 查看讨论线程和消息 |
| 1 | CREATE_MESSAGE | 在讨论线程中发送消息 |
| 2 | EDIT_OWN_MESSAGE | 编辑自己发送的消息（MVP 不实现） |
| 3 | APPROVE_PLAN | 审批执行计划 |
| 4 | REJECT_PLAN | 拒绝执行计划 |
| 5 | VIEW_KNOWLEDGE | 查看知识库文档（MVP 不实现） |
| 6 | EDIT_KNOWLEDGE | 编辑知识库文档（MVP 不实现） |
| 7 | MANAGE_MEMBERS | 管理项目成员（添加/移除） |
| 8 | MANAGE_AGENTS | 管理 Agent（创建/配置/暂停） |
| 9 | MANAGE_SPACE | 管理权限空间（MVP 不实现） |
| 10 | MANAGE_PROJECT | 管理项目设置（改名/描述） |
| 11 | ADMINISTRATOR | 绕过所有权限覆盖 |

```python
class HumanPermission(int, Enum):
    VIEW_DISCUSSION    = 1 << 0
    CREATE_MESSAGE     = 1 << 1
    EDIT_OWN_MESSAGE   = 1 << 2
    APPROVE_PLAN       = 1 << 3
    REJECT_PLAN        = 1 << 4
    VIEW_KNOWLEDGE     = 1 << 5
    EDIT_KNOWLEDGE     = 1 << 6
    MANAGE_MEMBERS     = 1 << 7
    MANAGE_AGENTS      = 1 << 8
    MANAGE_SPACE       = 1 << 9
    MANAGE_PROJECT     = 1 << 10
    ADMINISTRATOR      = 1 << 11

    @classmethod
    def all_bits(cls) -> int:
        return sum(member.value for member in cls)
```

### 1.2 Agent 权限位

| Bit | 权限名 | 说明 |
|-----|--------|------|
| 0 | QUERY_KNOWLEDGE | 查询知识库（MVP 不使用） |
| 1 | READ_DOCUMENT | 读取知识库文档（MVP 不使用） |
| 2 | POST_DISCUSSION | 在讨论线程中发布消息（总结 Agent 产出摘要） |
| 3 | GENERATE_PLAN | 生成执行计划（分解 Agent 产出） |
| 4 | GENERATE_CODE | 生成代码产出（执行 Agent 产出） |
| 5 | GENERATE_DOCUMENT | 生成文档产出（执行 Agent 产出） |
| 6 | REQUEST_APPROVAL | 发起审批请求 |
| 7 | WRITE_KNOWLEDGE | 写入知识库（MVP 不使用） |
| 8 | DELEGATE_TASK | 委派子任务（MVP 不使用） |
| 9 | NAVIGATE_LINKS | 导航知识库链接（MVP 不使用） |
| 10 | MANAGE_AGENTS | 管理其他 Agent（仅 Owner 可赋予） |

```python
class AgentPermission(int, Enum):
    QUERY_KNOWLEDGE    = 1 << 0
    READ_DOCUMENT      = 1 << 1
    POST_DISCUSSION    = 1 << 2
    GENERATE_PLAN      = 1 << 3
    GENERATE_CODE      = 1 << 4
    GENERATE_DOCUMENT  = 1 << 5
    REQUEST_APPROVAL   = 1 << 6
    WRITE_KNOWLEDGE    = 1 << 7
    DELEGATE_TASK      = 1 << 8
    NAVIGATE_LINKS     = 1 << 9
    MANAGE_AGENTS      = 1 << 10

    @classmethod
    def all_bits(cls) -> int:
        return sum(member.value for member in cls)
```

### 1.3 设计要点

- 人类和 Agent 权限位分开定义 — 权限含义不同（人类审批 vs Agent 产出）
- Bitmask 高效存储 — 单个 BIGINT 字段存储所有权限
- 后期添加权限位只需追加高位 — 不影响现有权限位值
- MVP 标注"不实现"的权限位 — bit position 已定义，但 MVP 不使用和检查

## 2. 角色 → 权限位映射

### 2.1 人类角色

| 角色 | 权限位 | Bitmask 值 | 说明 |
|------|--------|-----------|------|
| **Owner** | 所有权限 + ADMINISTRATOR | 0b111111111111 = 4095 | 项目创建者，拥有所有权限 |
| **Admin** | 除 ADMINISTRATOR 外的所有权限 | 0b1111111111 = 2047 | 管理项目设置、成员、Agent |
| **Member** | 查看、发言、审批、拒绝 | 0b11111 = 31 | 基础参与者权限 |
| **Viewer** | 只可查看 | 0b1 = 1 | 只读权限 |

```
Owner   = VIEW_DISCUSSION | CREATE_MESSAGE | EDIT_OWN_MESSAGE | APPROVE_PLAN | REJECT_PLAN | VIEW_KNOWLEDGE | EDIT_KNOWLEDGE | MANAGE_MEMBERS | MANAGE_AGENTS | MANAGE_SPACE | MANAGE_PROJECT | ADMINISTRATOR
        = 4095

Admin   = Owner & ~ADMINISTRATOR
        = 2047

Member  = VIEW_DISCUSSION | CREATE_MESSAGE | EDIT_OWN_MESSAGE | APPROVE_PLAN | REJECT_PLAN
        = 31

Viewer  = VIEW_DISCUSSION
        = 1
```

### 2.2 Agent 角色

> **与架构规格的对应关系**：0.1-arch.md 第 9.1 节定义了 4 种 Agent 角色（Assistant/Executor/Knowledge/Admin）。MVP 只实现 3 种，对应关系如下：
> - Summary Agent ↔ Agent Assistant（订阅讨论事件、生成总结）
> - Decompose Agent ↔ Agent Assistant 的任务分解能力（MVP 拆为独立角色）
> - Execute Agent ↔ Agent Executor（订阅审批事件、生成代码/文档）
> - Agent Knowledge ↔ MVP 不实现（Phase 2 知识库服务引入）
> - Agent Admin ↔ MVP 不实现（仅 Owner 可赋予，Phase 2 自定义 Agent 管理时引入）

| 角色 | 权限位 | Bitmask 值 | 说明 |
|------|--------|-----------|------|
| **Summary Agent** | POST_DISCUSSION | 0b100 = 4 | 在讨论中发布摘要 |
| **Decompose Agent** | POST_DISCUSSION, GENERATE_PLAN, REQUEST_APPROVAL | 0b1001100 = 76 | 产出执行计划 + 发起审批请求 |
| **Execute Agent** | GENERATE_CODE, GENERATE_DOCUMENT, REQUEST_APPROVAL | 0b1110000 = 112 | 产出代码/文档 + 发起审批请求 |

```
SummaryAgent  = POST_DISCUSSION
              = 4

DecomposeAgent = POST_DISCUSSION | GENERATE_PLAN | REQUEST_APPROVAL
               = 4 + 8 + 64 = 76

ExecuteAgent  = GENERATE_CODE | GENERATE_DOCUMENT | REQUEST_APPROVAL
              = 16 + 32 + 64 = 112
```

### 2.3 创建项目时的角色分配

- **创建者**：自动成为 Owner（roles = 4095）
- **添加成员时**：默认 Member（roles = 31），可指定其他角色
- **创建 Agent 时**：根据 agent_type 自动分配对应角色权限位

## 3. 权限计算逻辑

### 3.1 核心规则：Deny 永远胜过 Allow

```python
def compute_permissions(allow_bits: int, permission: int, deny_bits: int = 0) -> bool:
    """核心规则：Deny 永远胜过 Allow。

    同层级冲突时 deny 生效。

    Args:
        allow_bits: 角色/基础权限 bitmask
        permission: 要检查的单个权限位
        deny_bits: deny 覆盖 bitmask（MVP 不使用，默认 0）

    Returns:
        True — 有权限
        False — 无权限
    """
    if deny_bits & permission:
        return False   # Deny 胜过 Allow
    return bool(allow_bits & permission)
```

### 3.2 三层覆盖架构

| 层级 | 来源 | 说明 | MVP |
|------|------|------|-----|
| 第 1 层：项目基础权限 | 角色映射 | 人类角色/Agent角色 → bitmask | **实现** |
| 第 2 层：空间级覆盖 | 讨论/知识/执行空间 | allow/deny 覆盖项目基础权限 | 不实现 |
| 第 3 层：资源级覆盖 | 单个线程/文档/计划 | 独立权限设置 | 不实现 |

### 3.3 MVP 权限检查简化

MVP 只做项目级权限检查。计算流程：

```
1. 从 project_members 表读取 allow_bits（角色对应的 bitmask）
2. deny_bits = 0（MVP 不使用空间级/资源级覆盖）
3. compute_permissions(allow_bits, required_permission, deny_bits)
```

## 4. 权限检查在 API 层的实现

### 4.1 FastAPI Dependency Injection

```python
# permissions/compute.py

def require_permission(permission: int):
    """创建一个 FastAPI dependency，检查当前用户在当前项目中是否有指定权限。

    用法：
        @router.post("/plans/{id}/approve")
        async def approve_plan(
            plan_id: str,
            request: PlanApprove,
            user: User = Depends(get_current_user),
            _perm = Depends(require_permission(HumanPermission.APPROVE_PLAN)),
        ):
            ...
    """
    async def check_permission(
        user: User = Depends(get_current_user),
        project_id: str = None,  # 从路径参数提取
    ):
        member = await get_member(user.id, project_id)
        if not member:
            raise HTTPException(404, "Not a project member")

        if not compute_permissions(member.roles, permission):
            raise HTTPException(403, "Insufficient permissions")

        return True

    return check_permission
```

### 4.2 各端点权限要求

| 端点 | 所需权限位 |
|------|-----------|
| POST /auth/register | 无（公开） |
| POST /auth/login | 无（公开） |
| POST /projects | 无（任何认证用户可创建） |
| GET /projects | 无（只返回自己参与的项目） |
| GET /projects/{id} | 项目成员（VIEW_DISCUSSION） |
| POST /projects/{id}/members | MANAGE_MEMBERS |
| POST /projects/{id}/threads | CREATE_MESSAGE |
| GET /projects/{id}/threads | VIEW_DISCUSSION |
| POST /threads/{id}/messages | CREATE_MESSAGE |
| GET /threads/{id}/messages | VIEW_DISCUSSION |
| GET /projects/{id}/plans | VIEW_DISCUSSION |
| POST /plans/{id}/approve | APPROVE_PLAN |
| POST /plans/{id}/reject | REJECT_PLAN |
| GET /projects/{id}/outputs | VIEW_DISCUSSION |
| POST /outputs/{id}/approve | APPROVE_PLAN |
| POST /outputs/{id}/request-revision | CREATE_MESSAGE |
| POST /projects/{id}/agents | MANAGE_AGENTS |
| GET /projects/{id}/agents | VIEW_DISCUSSION |
| GET /projects/{id}/agents/{id}/status | VIEW_DISCUSSION |

### 4.3 权限检查流程

1. JWT → get_current_user → User 对象
2. URL path → project_id
3. project_members 表 → allow_bits (roles)
4. compute_permissions(allow_bits, required_permission)
5. 权限不足 → 403

**ADMINISTRATOR 权限位**绕过所有权限检查：

- Owner 角色（bitmask 包含 ADMINISTRATOR）自动通过所有权限检查
- compute_permissions 逻辑不变 — ADMINISTRATOR 位本身就是 allow_bits 中的一个位，如果角色包含此位，任何权限检查都会通过

## 5. Agent 权限与人类权限的关系

### 5.1 Agent 默认最少权限

Agent 的权限由 agent_type 自动分配，不需要手动配置。原则：

- Agent 只能做它声明能做的事（Summary Agent 只能发讨论消息，不能审批计划）
- Agent 不能执行人类操作（审批、管理成员等）
- Agent 产出需要人类审批（requires_approval=True）

### 5.2 Agent 不参与权限检查

MVP 中 Agent 不通过 REST API 操作。Agent 通过 EventBus 事件驱动：

- Agent 不调用 REST 端点 → 不需要 API 层权限检查
- Agent 权限位用于 **Phase 2 空间级覆盖** — 控制哪些 Agent 可以在哪些空间活跃
- MVP 中 Agent 权限位只用于 **展示**（GET /projects/{id}/agents 响应中的 roles 字段）

## 6. MVP 范围裁剪

| 功能 | MVP | Phase 2 |
|------|-----|---------|
| 权限层级 | 项目级 RBAC only | + 空间级覆盖 + 资源级覆盖 |
| Deny 覆盖 | 不实现（deny_bits=0） | 空间级/资源级 Deny 覆盖 |

| 权限审计日志 | 不做 | 权限变更事件记录 |
| 角色 | 4 种人类 + 3 种 Agent | 自定义角色 |
| 密码安全 | bcrypt | + 密码重置、2FA |
