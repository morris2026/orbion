## 1. EventBus Protocol — 抽象接口

所有业务模块只依赖 `EventBus` Protocol，不依赖具体实现。MVP 使用 `InProcessEventBus`，Phase 2 切换 `NATSEventBus` 时业务代码零改动。

### 1.1 Protocol 定义

```python
from typing import Protocol, Callable, Any, Awaitable

class EventBus(Protocol):
    """事件发布/订阅抽象接口。

    MVP 用 InProcessEventBus（进程内 asyncio 调度）。
    Phase 2 用 NATSEventBus（NATS JetStream 跨进程路由）。
    业务代码只依赖此接口，不依赖具体实现。
    """

    async def publish(self, event_type: str, payload: dict[str, Any]) -> None:
        """发布事件到总线。

        Args:
            event_type: 事件类型名称，如 "DiscussionMessageCreated"
            payload: 事件内容，必须包含 project_id 字段
        """
        ...

    def subscribe(
        self,
        event_type: str,
        handler: Callable[[dict[str, Any]], Awaitable[None]],
    ) -> str:
        """订阅事件类型，返回 subscription_id 用于取消订阅。

        Args:
            event_type: 要订阅的事件类型
            handler: 异步回调函数，接收事件 payload

        Returns:
            subscription_id: 用于 unsubscribe 时取消此订阅
        """
        ...

    def unsubscribe(self, subscription_id: str) -> None:
        """取消订阅。

        Args:
            subscription_id: subscribe 返回的 ID
        """
        ...
```

### 1.2 设计要点

- `publish` 接收 `event_type` + `payload` 而不是 `Event` 对象 — 保持接口与具体数据模型解耦
- `subscribe` 返回 `subscription_id` — 支持动态取消，Agent 启动/暂停时需要取消/恢复订阅
- `handler` 是 `AsyncCallable` — 所有业务处理都是异步的（FastAPI async 模式）
- MVP 不需要 `project_id` 参数 — 单进程内所有事件在同一空间，Phase 2 NATS 按 subject 路由

## 2. InProcessEventBus — MVP 实现

### 2.1 实现设计

```python
import asyncio
from collections import defaultdict
from uuid import uuid4

class InProcessEventBus:
    """进程内事件总线，MVP 使用。

    事件路由在同一进程内完成，零延迟，零外部依赖。
    所有 handler 通过 asyncio.create_task 调度，不阻塞发布者。
    """

    def __init__(self):
        self._handlers: dict[str, list[tuple[str, Callable]]] = defaultdict(list)

    async def publish(self, event_type: str, payload: dict[str, Any]) -> None:
        handlers = self._handlers.get(event_type, [])
        for sub_id, handler in handlers:
            asyncio.create_task(handler(payload))
            # create_task 保证 handler 不阻塞 publish
            # 如果 handler 失败，异常在 task 中捕获，不影响其他 handler

    def subscribe(
        self,
        event_type: str,
        handler: Callable[[dict[str, Any]], Awaitable[None]],
    ) -> str:
        sub_id = uuid4().hex
        self._handlers[event_type].append((sub_id, handler))
        return sub_id

    def unsubscribe(self, subscription_id: str) -> None:
        for event_type, subscribers in self._handlers.items():
            self._handlers[event_type] = [
                (sid, h) for sid, h in subscribers if sid != subscription_id
            ]
```

### 2.2 关键设计决策

1. **asyncio.create_task 调度 handler** — 不阻塞 publish 调用方。即使某个 handler 执行慢或失败，也不影响其他 handler 和发布者
2. **handler 异常不传播** — handler 失败时异常在 create_task 中记录日志，不传播到 publish。保证其他 handler 正常执行
3. **subscription_id 使用 uuid4** — 避免取消订阅时误删其他订阅
4. **defaultdict(list) 自动创建** — 订阅新事件类型时无需预注册

### 2.3 与 EventStore 的协作

`publish` 只做路由，不做持久化。持久化由调用方在 publish 前显式调用 EventStore.append：

```python
# 正确的事件发布流程（在 service.py 中）
await event_store.append(event)     # 先持久化
await event_bus.publish(event_type, payload)  # 再路由
```

持久化和路由分离，因为：

- 持久化是必须的（Event Store 不可变日志）
- 路由可能失败（handler 不可用），但不应影响持久化
- 投影更新是 handler 的职责之一，不需要 publish 时同步更新

## 3. NATSEventBus — Phase 2 实现

MVP 不实现此类，但定义好接口后 Phase 2 只需写一个新类即可替换。

### 3.1 Phase 2 实现设计

```python
import json
import nats

class NATSEventBus:
    """NATS JetStream 事件总线，Phase 2 多服务架构使用。

    事件按 subject 路由：orbion.events.{project_id}.{event_type}
    消费者组保证同类 Agent 只有一个实例处理每条事件。
    """

    def __init__(self, nats_url: str = "nats://localhost:4222"):
        self._url = nats_url
        self._nc: nats.NATS | None = None
        self._js: nats.JetStreamContext | None = None
        self._subs: dict[str, nats.Subscription] = {}

    async def connect(self):
        self._nc = await nats.connect(self._url)
        self._js = self._nc.jetstream()

    async def publish(self, event_type: str, payload: dict[str, Any]) -> None:
        subject = f"orbion.events.{payload['project_id']}.{event_type}"
        await self._js.publish(subject, json.dumps(payload).encode())

    def subscribe(
        self,
        event_type: str,
        handler: Callable[[dict[str, Any]], Awaitable[None]],
    ) -> str:
        sub_id = uuid4().hex
        subject = f"orbion.events.*.{event_type}"
        # Push consumer: NATS 自动将消息推送到 handler
        # Consumer group: 同类 Agent 只有一个处理每条消息
        self._subs[sub_id] = ...  # NATS push subscription
        return sub_id

    def unsubscribe(self, subscription_id: str) -> None:
        sub = self._subs.pop(subscription_id, None)
        if sub:
            await sub.unsubscribe()
```

### 3.2 切换成本

InProcessEventBus → NATSEventBus 的切换：

- **业务代码**：零改动。所有模块只依赖 EventBus Protocol
- **app/main.py**：改一行。`event_bus = InProcessEventBus()` → `event_bus = NATSEventBus(nats_url)`
- **docker-compose.yml**：加一个 NATS 服务
- **新增依赖**：nats-py（MIT 许可证）

## 4. EventStore — PostgreSQL 持久化

### 4.1 写端设计

```python
class EventStore:
    """事件持久化存储，PostgreSQL event_log 表。

    所有事件以不可变日志存储，支持按 correlation_id 和 project_id 查询。
    """

    def __init__(self, postgres_url: str):
        self._url = postgres_url
        self._pool: asyncpg.Pool | None = None

    async def connect(self) -> None:
        """创建数据库连接池。"""
        self._pool = await asyncpg.create_pool(self._url, min_size=2, max_size=10)

    async def close(self) -> None:
        """关闭连接池。"""
        if self._pool:
            await self._pool.close()

    async def append(self, event: Event) -> None:
        """追加事件到不可变日志。

        事件一旦写入不可修改或删除。
        """
        ...

    async def get_events_by_correlation(
        self, correlation_id: str, limit: int = 100
    ) -> list[Event]:
        """按 correlation_id 查询事件链。

        返回同一个 correlation_id 的所有事件，按时间排序。
        用于 Agent 组装上下文时获取历史事件。
        """
        ...

    async def get_events_by_project(
        self, project_id: str, event_type: str | None = None, limit: int = 50
    ) -> list[Event]:
        """按项目 ID 查询事件，可选按事件类型过滤。

        所有查询必须带 project_id（项目边界硬隔离）。
        """
        ...
```

### 4.2 event_log 表结构

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| event_id | UUID | PRIMARY KEY | 事件唯一标识 |
| project_id | VARCHAR(64) | NOT NULL | 项目边界硬隔离 |
| event_type | VARCHAR(64) | NOT NULL | 事件类型名称 |
| participant_id | VARCHAR(64) | NOT NULL | 发起者（人类用户ID 或 Agent ID） |
| participant_type | VARCHAR(8) | NOT NULL, CHECK IN ('human','agent') | 发起者类型 |
| payload | JSONB | NOT NULL DEFAULT '{}' | 事件内容 |
| correlation_id | UUID | NOT NULL | 跨事件追踪链 ID |
| causation_id | UUID | NULLABLE | 触发此事件的上级事件 ID |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() | 事件发生时间 |

### 4.3 索引

```sql
-- 项目边界查询（最频繁的查询模式）
CREATE INDEX idx_event_log_project ON event_log (project_id, created_at DESC);
-- correlation_id 链路追踪（Agent 上下文组装）
CREATE INDEX idx_event_log_correlation ON event_log (correlation_id, created_at);
-- 事件类型过滤（Agent 订阅特定事件类型时查询）
CREATE INDEX idx_event_log_type ON event_log (project_id, event_type);
```

## 4.4 信息分级存储策略

> 架构规格映射：第 4 章「Event Store & CQRS」第 4.4 节

架构规格（0.1-arch.md）定义了 4 级信息分级存储策略，控制 event_log 容量增长：

| 级别 | 内容 | MVP 实现 | Phase 2 |
|------|------|---------|---------|
| **0 — 全量永久** | 人类输入 + 元数据链 | event_log 全量存储，不可删除 | 不变 |
| **1 — 全量 + 大 payload 外置** | Agent 最终产出 | MVP 简化：payload 全量存 event_log JSONB | 大 payload 存对象存储，payload 字段放引用 ID |
| **2 — TTL 存储** | Agent 执行过程 | MVP 简化：不做 TTL，Agent trace 不进 event_log | agent_trace 表 + 30-90 天 TTL |
| **3 — 不进 event_log** | 系统操作 | SSE 连接/断开只进结构化日志，不占 event_log | 不变 |

MVP 简化策略：所有 Agent 最终产出（Level 1）payload 直接存 event_log JSONB，不做外置。Agent 执行过程（Level 2）MVP 不记录，只保留最终产出事件。SSE 连接管理（Level 3）只进 Python logging，不进 event_log。

## 5. CQRS 投影

### 5.1 MVP 4 个投影视图

| 投影视图 | 表名 | 更新触发事件 | 查询用途 |
|----------|------|------------|---------|
| 线程消息 | thread_messages | DiscussionMessageCreated, DiscussionSummaryGenerated | 前端讨论流展示 |
| 执行计划 | execution_plans | ExecutionPlanProposed, ExecutionPlanApproved, ExecutionPlanRejected | 前端计划卡片展示 |
| 任务产出 | task_outputs | TaskOutputGenerated, TaskOutputApproved, TaskOutputRevisionRequested | 前端 diff 预览 |
| 项目成员 | project_members | MemberAdded | 权限计算、成员列表 |

### 5.2 投影更新机制

投影由 EventBus 的 handler 触发更新：

```python
# events/projections.py 注册为 EventBus subscriber
class EventProjections:
    def __init__(self, event_bus: EventBus, postgres_url: str):
        self._bus = event_bus
        self._store = EventStore(postgres_url)
        # 注册订阅
        self._bus.subscribe("DiscussionMessageCreated", self._on_message_created)
        self._bus.subscribe("DiscussionSummaryGenerated", self._on_summary_generated)
        self._bus.subscribe("ExecutionPlanProposed", self._on_plan_proposed)
        self._bus.subscribe("MemberAdded", self._on_member_added)
        # ... 其他事件类型

    async def _on_message_created(self, payload: dict) -> None:
        # 更新 thread_messages 投影表
        await self._update_thread_messages(payload)

    async def _on_plan_proposed(self, payload: dict) -> None:
        # 更新 execution_plans 投影表
        await self._update_execution_plans(payload)
```

### 5.3 投影与 EventStore 的关系

- **EventStore**：不可变事件日志（写端）
- **投影表**：可变的聚合视图（读端）
- 前端查询读投影表（快速、结构化），不直接读 event_log
- 投影表数据可从 event_log 完全重建（丢投影表不影响核心数据）

## 7. correlation_id / causation_id 链路追踪

### 7.1 概念

- **correlation_id** — 串联整个协作链："讨论 → 总结 → 分解 → 审批 → 执行 → 审查"，同一个 ID
- **causation_id** — 记录直接因果关系：某事件的 causation_id 是触发它的上一级事件 ID

### 7.2 链路示例

```
事件 1: DiscussionMessageCreated
  correlation_id = "corr-001"
  causation_id = null（人类直接发起）

事件 2: DiscussionSummaryGenerated（总结 Agent 处理事件 1 后产出）
  correlation_id = "corr-001"（同一链路）
  causation_id = "evt-001"（因事件 1 而触发）

事件 3: ExecutionPlanProposed（分解 Agent 处理事件 2 后产出）
  correlation_id = "corr-001"
  causation_id = "evt-002"（因事件 2 而触发）

事件 4: ExecutionPlanApproved（人类审批）
  correlation_id = "corr-001"
  causation_id = "evt-003"

事件 5: TaskOutputGenerated（执行 Agent 处理事件 4 后产出）
  correlation_id = "corr-001"
  causation_id = "evt-004"
```

### 7.3 在 Agent 上下文组装中的使用

Agent 收到事件后，通过 correlation_id 从 EventStore 获取完整链路历史，组装 LLM prompt：

```python
# Agent 上下文组装
history = await event_store.get_events_by_correlation(payload["correlation_id"])
# history 包含整个协作链的所有事件，Agent 可以看到完整上下文
```

## 8. SSEChannel — ChannelPlugin 实现

### 8.1 ChannelPlugin 统一抽象

架构规格（0.1-arch.md 第 11 章）定义了 ChannelPlugin 抽象：所有外部入口统一为 ChannelPlugin 的不同实现。SSE 推送是 SSEChannel 的实现，IM 接入是 IMChannel 的实现（Phase 2）。

MVP 只实现 SSEChannel。Phase 2 加入 IMChannel 时，两者共享同一套 ChannelAdapter Protocol 和事件映射，业务代码零改动。

```python
class ChannelAdapter(Protocol):
    """Channel 插件抽象接口。
    
    SSEChannel 和 IMChannel 都实现此接口。
    业务代码只依赖此接口，不依赖具体 Channel 实现。
    """
    
    async def send_event(self, project_id: str, event_type: str, payload: dict) -> None:
        """将 Orbion 事件推送到外部通道。"""
        ...
    
    async def receive_event(self, external_event: dict) -> dict:
        """将外部通道事件翻译为 Orbion 事件 payload。"""
        ...
```

### 8.2 SSEChannel 设计

```python
class SSEChannel:
    """SSE 推送 Channel 实现（ChannelPlugin）。

    订阅 EventBus 的所有事件类型，维护按 project_id 分组的 SSE 连接池，
    将事件转化为 SSE 推送给前端。
    Phase 2 IMChannel 将实现同一套 ChannelAdapter Protocol，
    将 Orbion 事件翻译为 IM 平台消息格式推送。
    """

    def __init__(self, event_bus: EventBus):
        self._bus = event_bus
        self._connections: dict[str, list[StarletteSSEConnection]] = defaultdict(list)
        # 订阅所有需要推送的事件类型
        self._bus.subscribe("DiscussionMessageCreated", self._on_message)
        self._bus.subscribe("DiscussionSummaryGenerated", self._on_summary)
        self._bus.subscribe("ExecutionPlanProposed", self._on_plan_proposed)
        self._bus.subscribe("ExecutionPlanApproved", self._on_plan_approved)
        self._bus.subscribe("ExecutionPlanRejected", self._on_plan_rejected)
        self._bus.subscribe("TaskOutputGenerated", self._on_output)
        self._bus.subscribe("TaskOutputApproved", self._on_output_approved)
        self._bus.subscribe("TaskOutputRevisionRequested", self._on_revision)
        self._bus.subscribe("MemberAdded", self._on_member_added)

    def add_connection(self, project_id: str, connection: StarletteSSEConnection):
        """前端建立 SSE 连接时注册。"""
        self._connections[project_id].append(connection)

    def remove_connection(self, project_id: str, connection: StarletteSSEConnection):
        """前端断开 SSE 连接时移除。"""
        self._connections[project_id].remove(connection)

    async def send_event(self, project_id: str, event_type: str, payload: dict) -> None:
        """ChannelAdapter Protocol 实现：推送 SSE 事件。"""
        for conn in self._connections.get(project_id, []):
            await conn.send(event=event_type, data=json.dumps(payload))

    async def _on_message(self, payload: dict):
        await self.send_event(payload["project_id"], "message_created", payload)
```

### 8.3 SSEChannel 推送事件格式

| SSE event 名 | 来源事件类型 | 推送内容 |
|--------------|------------|---------|
| message_created | DiscussionMessageCreated | 消息内容（人类或 Agent） |
| summary_generated | DiscussionSummaryGenerated | 摘要卡片数据 |
| plan_proposed | ExecutionPlanProposed | 执行计划卡片数据 |
| plan_approved | ExecutionPlanApproved | 审批结果 |
| plan_rejected | ExecutionPlanRejected | 拒绝原因 + 修改建议 |
| output_generated | TaskOutputGenerated | 产出内容 + diff |
| output_approved | TaskOutputApproved | 产出审批结果 |
| revision_requested | TaskOutputRevisionRequested | 修改要求 + 问题列表 + 建议 |
| member_added | MemberAdded | 新成员信息（人类或 Agent） |
| agent_status_changed | Agent 内部状态变更 | Agent 运行状态 |

### 8.4 SSE 流端点

```
GET /events/stream?project_id={id}

认证：JWT（query param 或 header）
连接：长连接 SSE
断开：客户端关闭或超时
```
