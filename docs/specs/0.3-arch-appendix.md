## 2. 架构方案权衡分析

设计过程中评估了 3 种架构方案，最终选择方案 C（事件驱动 + Agent）。

### 2.1 方案 A：一体化架构

所有模块（讨论、AI、知识库）在同一个服务内。

**优点：** 开发最快，单服务部署，无服务间通信延迟，调试简单。
**缺点：** AI 和业务逻辑耦合，模型切换需改核心代码，知识库升级影响全系统，难以独立扩展 AI 计算。
**结论：** 模块边界模糊，后期扩展难，不适合 Orbion 的"人+Agent 双协作"需求。

### 2.2 方案 B：模块化微服务架构

核心服务 + AI 引擎独立服务 + 知识库独立服务，通过 API 直接调用。

**优点：** 模块边界清晰职责单一，AI 引擎可独立升级模型，知识库可独立演进，私有化/SaaS 切换只需改部署方式。
**缺点：** 初期需要设计 API 契约，服务间通信增加延迟，部署更复杂（3+ 服务）。
**结论：** 在简单性和扩展性之间取得平衡。但 Orbion 的核心模式"讨论 → AI 执行 → 审批"天然需要事件流串联，直接 API 调用不如事件驱动灵活。

### 2.3 方案 C：事件驱动 + Agent 架构

基于事件总线的松耦合架构，AI 以 Agent 形式接入。讨论产生事件 → Agent 订阅事件 → 生成执行计划 → 推回事件流。

**优点：**

- 极度灵活，可动态添加 Agent
- Agent 可用不同语言/框架和不同模型
- 天然支持未来多 Agent 协作
- 事件日志天然审计轨迹
- 人和 Agent 在事件系统中是对称的参与者

**缺点：**

- 事件流调试困难
- 状态管理复杂（需要 CQRS）
- MVP 有过度工程风险

**最终选择理由：**

1. Orbion 的核心定位是"人+Agent 双协作"，事件驱动天然让两者对称
2. Agent 动态增减只需订阅配置，不改核心代码
3. 可渐进式实现：Phase 1 只做最小骨架（开发量 ≈ 方案 B），架构已是事件驱动的
4. 业界验证：AutoGen 0.4 重写为事件驱动 Actor 模型，证明了方向的正确性

### 3.1 多 Agent 协作框架对比

|------|---------|-------------|-----------|--------|
| 架构风格 | 角色制，层级 Crew | 事件驱动，Actor 模型 | 图制，有状态工作流 | 事件驱动 + Agent 订阅 |
| Agent 定义 | role + goal + backstory | 对话参与者 | 图节点 + State schema | 订阅声明 + 模型配置 + 能力标签 |
| 通信方式 | 任务链式传递 | 异步消息传递 | 条件边 + 循环 | Event Bus 事件流 |
| 人机协作 | 弱（human_input flag） | 强（UserProxyAgent） | 强（checkpoint 中断/恢复） | 核心设计（审批式执行） |
| 状态管理 | 任务驱动，共享少 | 对话历史隐式状态 | 显式 State + reducer | Event Store + CQRS 投影 |
| 知识库 | 无内置 | 无内置 | 无内置 | 内置 Wiki + Context Injection |
| 多人协作 | 不支持 | 单用户 | 单用户 | 核心设计 |
| 与 Orbion 相似度 | 低 | 高（架构最接近） | 中（状态管理可借鉴） | — |

**借鉴决策：**

- AutoGen 0.4 的 Actor 模型与 Orbion 的 Agent-as-Event-Consumer 一致 → 验证方案 C 可行性，参考其消息传递协议设计
- LangGraph 的 checkpoint 中断/恢复 → 直接对应 Orbion 的审批式执行（Agent 产出后暂停，审批后恢复）
- CrewAI 的 role/goal/backstory → 用于 Orbion 的 Agent 注册声明格式，让 Agent 有明确的能力描述

### 3.2 AI 开发平台对比

| 维度 | Devin | Cursor | Replit Agent | Orbion |
|------|-------|--------|-------------|--------|
| 核心理念 | AI 自主开发 | AI 增强人类编码 | AI 全栈构建 | 人+Agent 协作执行 |
| 自主程度 | 全自动 | 人主导 | 半自主（有检查点） | 审批式执行 |
| 执行环境 | 独立 Docker sandbox | 本地 IDE | 云端容器 | Git 集成 + 审批后提交 |
| 多人协作 | 不支持 | 不支持（单用户IDE） | 多人实时编辑 | 讨论→AI→审批→执行 |
| 知识库 | 代码库记忆 | 代码库索引 | 包管理集成 | Wiki + Context Injection |
| Agent编排 | 单Agent内部循环 | 单Agent多步调用 | 规划→执行→调试循环 | 多Agent事件驱动链 |
| 部署模式 | SaaS | 本地安装 | SaaS | 私有化优先 |

**借鉴决策：**

- Devin 的 sandbox 环境设计 → Execution Service 可借鉴 Docker sandbox 思路隔离 Agent 代码执行
- Replit Agent 的"规划→执行→调试"三阶段 → 对应 Orbion 的"分解→执行→审批修改"流程
- Cursor 的代码库索引 + @引用上下文 → 知识库的 Context Injection 可借鉴其上下文组装策略

### 3.3 OpenClaw / Hermes-2 / Claude Code 对比

| 维度 | OpenClaw | Hermes-2 | Claude Code | Orbion |
|------|----------|----------|-------------|--------|
| 定位 | 多 Agent 框架 | LLM 模型（支持工具调用） | 单 Agent CLI 工具 | 多人+多Agent 协作平台 |
| 架构风格 | 4 层：记忆/规划/工具/协调 | 模型内置函数调用格式 | 工具循环 + MCP 扩展 | 事件驱动 + Agent 订阅 |
| 记忆系统 | 3 层：短期/长期/情景 | 无 | 会话级 + 项目级 CLAUDE.md | Event Store 投影 + Wiki 知识 + memory.md |
| 规划 | HTN 分层任务网络 | 无 | 隐式 | 显式（分解 Agent 产出 ExecutionPlan） |
| 工具使用 | 动态注册 + 组合链 | function_call 格式 | 内置工具 + MCP 扩展协议 | Agent Skill 声明 + 知识检索接口 |
| 权限 | 无 | 无 | 分级：自动允许 / 需确认 | 项目级 RBAC + 空间级覆盖 + Agent 能力约束 |
| 人机协作 | 弱 | 无 | 强 | 核心设计（审批式执行） |
| 扩展性 | Python 插件 | 兼容多个框架 | MCP 协议（标准化） | 声明式 Agent 注册 + 抽象接口 |
| 与 Orbion 相似度 | 中 | 低（是模型不是框架） | 高 | — |

**借鉴决策：**

从 OpenClaw 借鉴：

- 3 层记忆可映射到 Orbion：短期→Event Store 事件流，长期→Wiki 知识库，情景→memory.md + backstory
- HTN 规划思路可借鉴分解 Agent 的任务分解策略
- 共享记忆验证了 Orbion Event Store + Wiki 作为跨 Agent 共享"记忆"的方向

从 Claude Code 借鉴（最核心的参考）：

- "工具优于指令" → Agent 智能来自 Skill 组合和 KnowledgeRetriever，不靠 backstory 灌知识
- MCP 思路 → Orbion 设计类似的"Agent Tool Protocol"，Agent 通过声明式注册工具能力
- 分级权限 → 低风险自动允许，中风险项目级控制，高风险必须人类审批
- 子 Agent 委派 → 执行 Agent 可派生并行子 Agent 处理独立任务
- 简单循环优于复杂编排 → MVP 只做 2-3 个 Agent + 简单事件链

从 Hermes-2 借鉴：

- 结构化工具调用格式 → Agent 产出事件 payload 用结构化 JSON
- 模型适配器统一格式 → ModelAdapter 用标准化函数调用格式确保不同模型兼容

### 3.4 Anthropic "Building Effective Agents" 模式映射

| Anthropic 模式 | Orbion 映射 |
|----------------|------------|
| Prompt Chaining | 讨论→总结→分解→执行（事件链串联） |
| Routing | AgentScheduler 根据事件类型路由到对应 Agent |
| Parallelization | ExecutionPlanApproved 后多任务分派给多个执行 Agent 并行 |
| Evaluator-Optimizer | 审批修改循环：Agent 产出→人类审查→RevisionRequested→Agent 重做 |
| Orchestrator-Worker | 分解 Agent 统筹（Orchestrator）→ 执行 Agent 并行（Worker） |

## 4. 知识库方案权衡 — RAG vs Context Injection

### 4.1 方案 A：RAG

流程：文档 → 切片 → Embedding → 写入向量库 → 查询 → 向量检索 top-K → 拼凑 chunk → 注入 prompt

**优点：** 适合大规模语料（>1M token），知识更新频繁时无需重新加载上下文，成本可控（只检索相关 chunk）。
**缺点：** 切片破坏文档结构关系，向量检索是概率性的不可控，需要额外向量库基础设施（Qdrant/Chroma），Embedding 模型选择和维护成本，多跳推理能力弱，检索过程不可追溯。

### 4.2 方案 B：Context Injection

流程：Wiki 结构 → 标签/链接/层级导航 → 选定相关文档 → 序列化为结构化 prompt → 注入 Agent 上下文窗口

**优点：**

- 保留文档间关系（链接、层级）
- 确定性检索，可复现
- 无需向量库基础设施（只有 PostgreSQL + Markdown 解析）
- 零 Embedding 成本
- 天然支持多跳推理（沿链接跳转）
- 人类可控 — 用户通过标签和链接主动组织知识
- 审计清晰 — Agent 获取了哪些文档、沿哪些链接跳转，全部记录在事件流中
- 私有化部署简单 — 只需 PostgreSQL

**缺点：**

- 上下文窗口限制 — Claude 200K 可放约 10-15 篇完整文档，超过需要分层裁剪
- 知识库组织质量依赖 — 懒散的 Wiki = 低质量上下文
- 无自动发现 — 如果用户没建标签/链接，Agent 就找不到相关文档
- 大知识库裁剪复杂 — 超过 500 篇文档需要核心层/关联层/外围层策略

**选择理由：**

1. 用户明确表示不想用 RAG
2. 2025 业界趋势：CAG（Context-Augmented Generation）正在取代传统 RAG
3. Claude 200K / Gemini 2M 大窗口让直接注入成为可能
4. Obsidian 社区已实践 Wiki 结构 → 结构化 prompt 的方式
5. 私有化部署只需 PostgreSQL，不需要额外向量库服务
6. 通过 KnowledgeRetriever 抽象接口，未来可替换实现而不影响 Agent 代码

## 5. Agent 记忆方案权衡 — 统一知识库 vs 分离层次化 Markdown

### 5.1 方案 A：知识库统一管理 Agent 记忆

将 Agent 经验记忆存为 Wiki 中的特殊文档（如 `[[Agent:总结助手-经验]]`）。

**优点：** 零额外基础设施，复用 Wiki 和 PostgreSQL，人类可审查。
**缺点（严重）：**

- 知识污染 — Agent 经验日志混入人类知识库，降低 Wiki 质量
- 生命周期冲突 — 知识是长期维护的，Agent 记忆是临时可重置的
- 结构冲突 — 知识用 Obsidian 双向链接，记忆用层次化标题
- 访问模式不同 — 知识是所有 Agent + 人类共享检索，记忆是特定 Agent 的私有状态
- 质量标准不同 — 知识要求人类审核高质量，Agent 记忆是机器生成可能含错误

### 5.2 方案 B：分离层次化 Markdown

Agent 记忆独立存储为层次化 memory.md 文件，与知识库完全分离。

**优点：**

- 职责分离 — 知识库管知识，Agent 记忆管状态
- 检索精度高 — 知识库只含高质量知识，不被经验日志干扰
- 幽觉率低 — 程序性规则不被当作情景事实检索（研究数据：幻觉率 -18%）
- 生命周期独立 — Agent 记忆可重置、清理、版本管理
- 人类可审查且不污染 — memory.md 在独立存储中，不影响知识库
- Diff 可版本管理 — Markdown 文件可 git 版本化
- 与 Claude Code CLAUDE.md 模式一致 — 所有层级用同一文件名 memory.md，路径表达范围

**缺点：**

- 需要独立存储（但仍是 PostgreSQL 或文件系统，不是向量库）
- 跨存储同步 — Agent 经验提炼的规律需要手动同步到知识库
- 实现复杂度略高

**选择理由：**

1. 研究数据：分离存储准确率 +23%，幻觉率 -18%（ArXiv 2025 论文）
2. 用户反馈：知识库应只做知识管理，不管理 Agent 记忆
3. Claude Code 用独立 CLAUDE.md 管理行为指令，知识通过工具获取 — 验证了分离方向
4. MemGPT/Letta 用独立分层记忆，知识库是外部资源
5. LangGraph 用独立 Checkpoint Store，知识通过 Tool 获取

## 6. 权限模型权衡 — 各平台对比

| 平台 | 层级结构 | 粒度 | 复杂度 | 与 Orbion 适配度 |
|------|---------|------|--------|-----------------|
| Discord | Server → Category → Channel | 极高（bitmask ~50权限） | 中高 | 高 |
| GitHub | Org → Repo → Branch | 中（开发者聚焦） | 中 | 中 |
| Jira | Global → Project → Issue | 极高（完全可配置） | 高 | 低（太复杂） |
| Linear | Workspace → Team → Project | 低（极简） | 低 | 中 |
| Notion | Workspace → Page → Sub-page | 中（继承式） | 低中 | 低（只适合文档） |
| Slack | Grid → Workspace → Channel | 低中（通信聚焦） | 低中 | 低（太简单） |

**借鉴决策：** Discord 的三层覆盖模型最适合 Orbion，因为：

- 三层覆盖（项目→空间→资源）映射 Orbion 的权限需求
- Deny 胜过 Allow 对 Agent 场景尤其重要（默认限制 Agent）
- Bitmask 权限位高效存储
- 但补充了 Discord 缺少的：项目结构、审批流权限、Agent 角色体系、ABAC 上下文约束

## 7. 2025 事件驱动 AI 架构趋势验证

设计过程中搜索到的业界趋势，验证了方案 C 的方向正确性：

1. AutoGen 0.4 重写为事件驱动 Actor 模型 — Microsoft 从对话式改为事件驱动，与 Orbion 方案 C 几乎一致
2. CQRS + Event Sourcing 被引入 AI — 2025 多篇实践提出将 Agent 决策存储为不可变事件流，与 Orbion 的 Event Store 一致
3. Projection → Prompt 模式 — 从事件流投影构建 LLM 上下文窗口，这正是 Orbion Agent 获取上下文的方式
4. Saga 模式用于多步 LLM 工作流 — 与 Orbion 的多 Agent 链式协作对应
5. Agent Trajectory Branching — 在事件节点分叉探索不同路径，可用于"Agent 产出多个候选方案供人类选择"
6. CAG 正在取代传统 RAG — 精心组织的上下文比概率检索更准确，验证了 Context Injection 方向
7. 长上下文窗口（Claude 200K / Gemini 2M） — 让直接注入整篇文档成为可能，不再需要切片
