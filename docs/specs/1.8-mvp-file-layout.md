## 1. 设计原则

1. **项目是第一维度** — Orbion 的协作模型是"多人 + 多 Agent 围绕一个项目协作"，与 GitHub repo-first 一致，文件系统布局必须对齐协作模型
2. **单根目录 = 单备份单元** — 一个 ORBION_DIR 即完整恢复，无散落数据
3. **数据库存结构化索引，文件系统存大块内容** — 元数据、关系、权限在 PostgreSQL；Agent 记忆、Git 产出、知识库文档、大 payload 在文件系统
4. **环境变量覆盖路径** — ORBION_DIR 控制根目录，开发/生产/容器内用不同值，目录结构不变

## 2. 业界参考与选型

| 项目 | 根目录 | 第一维度 | 适用场景 | 与 Orbion 的差异 |
|------|--------|----------|----------|-----------------|
| OpenClaw | `~/.openclaw/` | Agent（per-agent 目录隔离） | 本地 CLI，单用户多 Agent | Orbion 是多人协作平台，用户不是 Agent |
| Myco | `MYCO_STATE_DIR` → `/data` | 用户（`/wks/<user>/<session>/`） | 容器部署，多租户共享服务器 | 用户优先导致协作时 admin 的编辑落在 owner 目录，项目级产出被困在用户路径下 |
| GitHub | — | 仓库（repo-first） | 多人围绕仓库协作 | Orbion 的协作模型与 GitHub 最接近 |

**Orbion 选择项目优先的原因：**

- 协作模型对齐 — 多人 + 多 Agent 围绕项目工作，项目是协作的物理边界
- 避免用户优先的协作矛盾 — Myco 的用户优先布局导致 admin 编辑落在 owner 目录、项目级 _myco_/ 被困在用户路径下、访问控制逻辑需要多处 workaround
- 与数据库设计一致 — PostgreSQL 已按 project_id 做硬隔离，文件系统同步按项目组织
- 符合 CI/CD 习惯 — GitHub Actions `/home/runner/work/<repo>/`、Jenkins `/var/lib/jenkins/workspace/<job>/` 都是项目优先

## 3. 目录结构

### 3.1 完整目录树

```
ORBION_DIR/                                     # 默认 ./data
│
├── orbion.json                                  # 平台配置（非密钥）
├── .env                                         # 密钥（JWT secret、API keys 等）
├── memory.md                                    # 平台级 Agent 记忆（所有项目、所有 Agent 的默认行为）
│
├── agents/                                      # Agent 模板（新建项目时从此拷贝）
│   └── <agentType>/                             # 按 Agent 类型组织
│       ├── declaration.json                     # Agent 声明（role/goal/backstory/model_config 等）
│       └── memory.md                            # 默认记忆模板
│
├── projects/<projectId>/                        # 项目是第一维度
│   ├── memory.md                                # 项目级 Agent 记忆
│   ├── agents/<agentType>/                      # Agent 级记忆
│   │   └── memory.md
│   ├── threads/<threadId>/                      # 线程是协作的具体场所
│   │   └── attachments/                         # 线程讨论附件（Phase 2）
│   ├── repo/                                    # 仓库容器（内含 git clone 的项目）
│   │   └── <repoName>/                         # 单个 git 仓库（MVP 一个项目一个仓库）
│   │       ├── .git/
│   │       └── ...
│   ├── knowledge/                               # 知识库文档（Phase 2）
│   │   └── <docId>.md
│   └── outputs/                                 # 大 payload 外置存储（信息分级 Level 1）
│       └── <outputId>.json
│
├── users/<userId>/                              # 用户级空间
│   ├── profile.json                             # 用户身份偏好
│   └── credentials.enc                          # Git 凭据（Fernet 加密，内含凭据列表）
│
├── logs/                                        # 持久日志
│   └── audit/                                   # 审计日志
│
└── temp/                                        # 临时文件（不备份）
    └── sandbox/                                 # 沙箱工作区
```

| 路径 | 用途 | 持久化 | 备份 |
|------|------|--------|------|
| `orbion.json` | 平台非密钥配置 | 是 | 是 |
| `.env` | 密钥文件（JWT secret、API keys 等） | 是 | 是 |
| `memory.md` | 平台级 Agent 默认行为记忆 | 是 | 是 |
| `agents/<type>/declaration.json` | Agent 声明模板（role/goal/backstory/model_config） | 是 | 是 |
| `agents/<type>/memory.md` | Agent 记忆模板（项目注册 Agent 时拷贝此文件） | 是 | 是 |
| `projects/<id>/memory.md` | 项目级 Agent 记忆 | 是 | 是 |
| `projects/<id>/agents/<type>/memory.md` | Agent 类型级记忆 | 是 | 是 |
| `projects/<id>/threads/<tid>/` | 线程目录（MVP 不创建，线程数据在数据库） | 是 | 是 |
| `projects/<id>/threads/<tid>/attachments/` | 线程讨论附件 | 是 | 是 |
| `projects/<id>/repo/<name>/` | Git 仓库（MVP 单仓库，Phase 2 多仓库） | 是 | 是 |
| `projects/<id>/knowledge/` | 知识库 Markdown 文档 | 是 | 是 |
| `projects/<id>/outputs/` | 大 payload 外置（task_outputs.id 为文件名，event_log 只存引用 ID） | 是 | 是 |
| `users/<id>/profile.json` | 用户偏好 | 是 | 是 |
| `users/<id>/credentials.enc` | 用户 Git 凭据（Fernet 加密，内含凭据列表） | 是 | 是 |
| `logs/` | 运行日志 | 是 | 可选 |
| `temp/` | 临时文件 | 否 | 否 |

## 4. 第一维度选择：项目优先

为什么不选用户优先或 Agent 优先：

**用户优先的问题（Myco 的教训）：**

```
/wks/<user>/<session>/<project>/        ← 用户优先
```

- Admin B 加入 A 的项目，B 的编辑落在 A 的目录下，B 无法独立备份
- 项目级产出（_myco_/）被困在用户路径下，多人协作时产生独立副本，只有通过 git push/pull 才能收敛
- 访问控制逻辑复杂化——路径维度和权限维度混在一起

**Agent 优先的问题（OpenClaw 的做法）：**

```
agents/<agentId>/agent/                  ← Agent 优先
```

- 适合单用户多 Agent 的 CLI 场景，不适合多人协作平台
- Orbion 中 Agent 是项目成员（project_members 表中 type='agent'），不是独立租户

**项目优先的优势：**

```
projects/<projectId>/                   ← 项目优先
```

- 与协作模型对齐——多人 + 多 Agent 围绕项目工作
- 与数据库设计一致——PostgreSQL 按 project_id 硬隔离
- 删除项目时直接删除整个目录树，无需跨用户目录清理
- Git repo 自然隔离——每个项目独立仓库，产出不混在一起

## 5. 配置与环境变量

```python
class Settings(BaseSettings):
    # 单一根路径控制，替代原有的 repo_path + memory_base_path
    root_dir: str = "./data"

    model_config = SettingsConfigDict(env_prefix="ORBION_")
```

环境变量控制根路径，内部结构固定：

| 环境 | ORBION_ROOT_DIR | 说明 |
|------|------------|------|
| 本地开发 | `./data` | 项目根目录下 |
| 容器内 | `/data` | bind-mount 宿主机目录 |
| 生产自定义 | `/var/lib/orbion` | FHS 兼容 |

Why: pydantic-settings 的 `env_prefix="ORBION_"` 将字段 `root_dir` 映射为 `ORBION_ROOT_DIR`，与 `ORBION_JWT_SECRET`、`ORBION_EVENT_STORE` 等一致

派生路径由代码计算，不暴露为配置项：

```python
@property
def projects_dir(self) -> Path:
    return Path(self.root_dir) / "projects"

@property
def platform_memory_path(self) -> Path:
    return Path(self.root_dir) / "memory.md"

def project_dir(self, project_id: str) -> Path:
    return self.projects_dir / project_id

def project_memory_path(self, project_id: str) -> Path:
    return self.project_dir(project_id) / "memory.md"

def agent_memory_path(self, project_id: str, agent_type: str) -> Path:
    return self.project_dir(project_id) / "agents" / agent_type / "memory.md"

def thread_dir(self, project_id: str, thread_id: str) -> Path:
    return self.project_dir(project_id) / "threads" / thread_id

def project_repo_path(self, project_id: str, repo_name: str) -> Path:
    return self.project_dir(project_id) / "repo" / repo_name

def output_payload_path(self, project_id: str, output_id: str) -> Path:
    return self.project_dir(project_id) / "outputs" / f"{output_id}.json"
```

## 6. 与现有设计的映射

### 6.1 替代原有的独立路径配置

原有两个独立路径合并为一个 `root_dir`，子路径按规则派生。实现时直接替换 `config.py` 中的 `repo_path` 和 `memory_base_path`。

| 原有配置 | 原默认值 | 新位置 |
|----------|----------|--------|
| `repo_path` | `./repo` | `root_dir/projects/<id>/repo/` |
| `memory_base_path` | `./data/memory` | `root_dir/memory.md` + `root_dir/projects/<id>/` |

原有两个独立路径合并为一个 `root_dir`，子路径按规则派生。

### 6.2 Agent 记忆路径对齐

原设计（`1.6-mvp-agent-runtime.md` 8.1 节）中的层次结构：

```
memory.md
project/{id}/memory.md
project/{id}/agents/{agent_type}/memory.md
```

新设计中完整对齐，只是增加了 `ORBION_DIR/` 前缀，以及项目下的其他子目录（repo/、knowledge/、outputs/）。

任务级记忆不做——链路上下文由 Event Store 按 correlation_id 提供，memory.md 只解决跨 correlation_id 的持久偏好，三层（平台→项目→Agent）足够。

### 6.3 Git 仓库按项目隔离

原有 `repo_path` 是全局单仓库，所有项目的产出混在一起。新设计中每个项目独立 `repo/` 目录：

- `repo/` 是仓库容器，内含一个或多个 `git clone` 的项目
- MVP 一个项目对应一个仓库，Phase 2 支持多仓库（如前后端分开的 monorepo）
- 产出 commit 写入对应仓库子目录
- 删除项目时连带删除整个 `repo/` 容器
- 不同项目可以配置不同的 remote（Phase 2 push 支持）

```
projects/<id>/repo/
├── orbion/                  # MVP：单个仓库即足够
│   ├── .git/
│   ├── src/
│   └── ...

# Phase 2：多仓库场景
projects/<id>/repo/
├── frontend/
│   ├── .git/
│   └── ...
├── backend/
│   ├── .git/
│   └── ...
```

### 6.4 Git 凭据存储

clone 私有仓库需要认证凭据。凭据按用户存储，clone 时根据 URL 自动匹配，用户无需手动选择。

**存储路径**：`users/<userId>/credentials.enc`

每用户一个文件，Fernet 对称加密（AES-128-CBC + HMAC-SHA256），per-user key 由 HKDF 从 `jwt_secret` 派生：

```python
HKDF(
    algorithm=SHA256(),
    length=32,
    salt=b"orbion-credential-encryption-key",
    info=user_id.encode(),   # per-user key
).derive(jwt_secret.encode())
```

每个用户的加密密钥不同，即使 `jwt_secret` 泄露，仍需逐一派生才能解密各用户凭据。

解密后内容为 JSON 数组：

```json
[
  {
    "id": "uuid",
    "type": "github",
    "name": "我的 GitHub",
    "token": "ghp_xxxx",
    "created_at": "2026-06-16T..."
  }
]
```

**凭据类型**：

| type | 显示名 | 字段 | URL 匹配规则 |
|------|--------|------|-------------|
| `github` | GitHub | token | URL host 含 `github.com` |

设计约束：

- 类型是后端枚举，用户创建时从预定义类型中选择，不允许自由填写
- 同一类型可存多条凭据（如多个 GitHub account），匹配时取最新
- 元数据（type/name/host）仅在解密后可见，文件本身整体加密，无法从文件推断凭据数量和类型
- 文件不存在时视为空列表
- 目录权限 700，文件权限 600

**clone 时自动匹配**：

`CredentialService.match_credential(user_id, url)` 从 URL 提取 host，遍历用户凭据列表匹配类型：

1. URL host 含 `github.com` → 匹配 type=`github`
2. 无匹配 → 无认证 clone（公开仓库或 SSH）

匹配到凭据后，通过 git credential helper 注入认证，而非拼入 URL：

**Credential helper 方案**：

git credential helper 是 git 原生机制，所有 git 操作（clone/pull/push）需要认证时自动调用。凭据永远不出现在 URL 或 `.git/config` 中。

实现方式：

1. `CredentialService` 写一个临时 Python 脚本（`credential-helper-{user_id}.py`），git 需要认证时调用它
2. 脚本接收到 git 传入的 protocol + host，从加密文件解密凭据，匹配后输出 username + password
3. clone/pull/push 时通过 `git -c credential.helper=...` 指定该 helper
4. clone 完成后，在 repo 级别配置 `credential.helper`，后续 pull/push 也自动走该 helper

helper 脚本核心逻辑：

```
# git 调用时传入：
protocol=https
host=github.com

# helper 输出：
username=x-access-token
password=ghp_xxxx
```

安全要点：

- helper 脚本本身不含凭据，仅包含"根据 host 从加密文件解密匹配凭据"的逻辑
- 凭据仅在 git 请求认证时按需解密，不缓存明文
- URL 中无凭据，`.git/config` 中无凭据，`.git/credentials` 中无凭据

**API 端点**（挂载在 `/users/me/credentials`）：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/` | 列出凭据（返回元数据，不含 token） |
| POST | `/` | 创建凭据 |
| DELETE | `/{credential_id}` | 删除凭据 |

**前端入口**：新建项目对话框的"仓库"输入框旁增加钥匙图标，点击打开凭据管理对话框。clone 时后端自动匹配，前端无需选择凭据。

### 6.5 Agent 模板与项目实例

平台级 `agents/<type>/` 存放 Agent 模板，项目级 `agents/<type>/` 存放从模板拷贝后的项目实例：

```
# 平台级模板（只读，供拷贝）
agents/summary/declaration.json     → Agent 声明（role/goal/backstory/model_config）
agents/summary/memory.md            → 默认记忆模板

# 项目级实例（可自定义，与模板独立）
projects/<id>/agents/summary/memory.md  → 从模板拷贝后可自定义
```

模板机制：

1. **注册 Agent 时** — 从 `agents/<type>/memory.md` 拷贝到 `projects/<id>/agents/<type>/memory.md`
2. **项目自定义** — 项目级 memory.md 可独立修改，不影响模板和其他项目
3. **模板更新** — 修改平台级模板只影响新注册的 Agent，已注册的 Agent 不受影响
4. **MVP 内置 3 个模板** — summary、decompose、execute，Phase 2 支持自定义 Agent 类型
5. **MVP 项目级只有 memory.md** — 项目级 Agent 声明完全继承模板，不需要 `declaration.json`；Phase 2 支持自定义时，项目级 `agents/<type>/` 下可增加 `declaration.json` 覆盖模板默认值

## 7. 数据库与文件系统的职责划分

| 数据类型 | 存储位置 | 原因 |
|----------|----------|------|
| 事件日志 | PostgreSQL `event_log` | 不可变、需索引、按 project_id 查询 |
| 投影视图 | PostgreSQL 各投影表 | 需关联查询、实时更新 |
| 用户认证 | PostgreSQL `users` 表 | 需事务、索引、密码哈希 |
| 成员权限 | PostgreSQL `project_members` 表 | 需 bitmask 计算、关联查询 |
| Agent 记忆 | 文件系统 `memory.md` | Markdown 格式、Agent 读写、层次化加载 |
| Git 产出 | 文件系统 `repo/` | Git 操作天然基于文件系统 |
| 知识库文档 | 文件系统 `knowledge/` | Markdown 编辑、双向链接解析 |
| 大 payload | 文件系统 `outputs/` | 避免 JSONB 膨胀、event_log 只存引用 ID |
| 用户偏好 | 文件系统 `profile.json` | 轻量配置，无需索引 |
| Git 凭据 | 文件系统 `users/<id>/credentials.enc` | 加密存储、按用户隔离、clone 时按需解密 |

## 8. 与 Myco 目录规划对比

Myco（容器化 AI 协作平台）的目录规划与 Orbion 有显著差异，以下是逐项对比中发现的待规划项：

### 8.1 需要补充的目录

| 目录 | 用途 | 规划阶段 |
|------|------|----------|
| `users/<id>/credentials.enc` | Git PAT 等凭据加密存储 | MVP，clone 私有仓库时需要 |
| `projects/<id>/threads/<tid>/media/` | 线程讨论中的图片、文件等媒体附件 | Phase 2，替代当前 `attachments/` 更明确 |
| `extensions/` | 插件/Channel/Skill 安装 | Phase 2，Channel 插件、Skill 市场 |
| `locks/` | 文件锁（多 Agent 并发写文件） | Phase 2，多 Agent 并发执行时需要 |

### 8.2 已有等价实现（目录结构不同但功能已覆盖）

| Myco | Orbion | 说明 |
|------|--------|------|
| `auth-sessions.json` | PostgreSQL `users` 表 + JWT | 认证令牌存数据库 |
| `sessions.json` | PostgreSQL `threads` 表 | 会话/线程概念存数据库 |
| `allowed-github-users.txt` | PostgreSQL admin 审批流程 | 用户准入存数据库 |
| `transcripts/` | `event_log` + `thread_messages` 投影表 | 会话记录存数据库 |
| `_myco_/` (随代码走) | `projects/<id>/` 下记忆文件 | 见 8.3 |

### 8.3 项目记忆：随状态走而非随代码走

Myco 将项目记忆放在 git repo 内的 `_myco_/` 目录，git clone 即带走。Orbion 选择将记忆放在 `root_dir/projects/<id>/` 下，与 repo 分离。

设计决策：

- **项目级记忆跟着项目走，而非跟着代码走** — `projects/<id>/memory.md` 是项目级全局记忆，多线程共享，所有协作者在平台内看到同一份
- **repo 只存代码产出** — 记忆和代码物理隔离，避免敏感信息误提交到 git，避免多人各自 clone 导致记忆副本不一致
- **新队友通过平台获取上下文** — 不依赖 git clone 带走记忆，而是在 Orbion 平台内参与项目即拥有完整上下文

这类似 Claude Code 的模式：项目目录下的 `CLAUDE.md` 是项目级约定，而非 git 追踪的记忆文件。

### 8.4 不需要的目录

| Myco 目录 | 原因 |
|-----------|------|
| `home/` (SDK home) | Orbion 自有 Agent Runtime，不依赖外部 SDK |
| `delivery-queue/` | Orbion 走 Event Bus 而非消息投递队列 |
| `cron/jobs.json` | 事件驱动架构，无定时任务场景 |
| `completions/` | Web 应用无 CLI 补全 |
| `tui/` | Web 应用无 TUI |

## 9. 部署映射

### Docker Compose 生产部署

```yaml
services:
  backend:
    volumes:
      - /home/orbion/data:/data           # ORBION_DIR=/data
    environment:
      ORBION_DIR: /data
```

### 目录权限

| 目录 | 属主 | 权限 | 说明 |
|------|------|------|------|
| `ORBION_DIR/` | orbion | 750 | 根目录仅属主和组可访问 |
| `.env` | orbion | 600 | 密钥仅属主可读写 |
| `projects/<id>/repo/` | orbion | 755 | Git 操作需要读写 |
| `projects/<id>/outputs/` | orbion | 750 | 产出物敏感度由项目权限控制 |
| `users/<id>/` | orbion | 700 | 用户数据严格隔离 |
| `users/<id>/credentials.enc` | orbion | 600 | 凭据文件仅属主可读写 |
| `temp/` | orbion | 755 | 临时文件无需保护 |

## 10. 备份与恢复

单根目录即完整备份单元：

```bash
# 停止服务后热备
pg_dump orbion | gzip > backup_$(date +%Y%m%d).sql.gz
tar czf data_$(date +%Y%m%d).tar.gz -C /home/orbion/data .

# 恢复
tar xzf data_20260612.tar.gz -C /home/orbion/data
psql orbion < backup_20260612.sql
```

备份排除项：`temp/`（可重建）、`logs/`（非关键数据）。
