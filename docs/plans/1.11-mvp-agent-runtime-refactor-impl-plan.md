# Orbion Agent Runtime 重构 实施计划

## 策略

**数据模型与加密底座先行，逐层向上构建 ModelAdapter → Skill → AgentRuntime → 事件调度 → 失败恢复 → Critic → 审批传播 → 可观测性**：先建全部新表 + AES-GCM 加密 + 用户模型配置 API，再实现双 SDK ModelAdapter 与 Skill 系统，然后是 AgentRuntime 核心 dispatch/chat，接着事件驱动调度、失败恢复、Critic、审批与变更传播，最后是 token 统计、可观测性与运维收尾。

选择理由：

- 加密底座（AES-GCM）和 UserModel 表是 AdapterFactory 的硬依赖，必须先就绪
- ModelAdapter Protocol 是 dispatch 的核心依赖，要在 AgentRuntime 之前实现
- Skill 系统与 AdapterFactory 可并行，但 Skill 调用发生在 dispatch 内部循环，需先于 AgentRuntime
- AgentRuntime.dispatch 是事件消费方，必须在 EventBus/scheduler 路由之前
- 失败恢复（reconcile/事务补偿/优雅停机）依赖 agent_runs 表和 dispatch 流程
- Critic 是 task completed 后的可选触发，依赖 dispatch + agent_runs.run_kind=critic
- 审批合并依赖 artifact 状态机 + WorktreeService.merge（已实现）+ 变更传播依赖 based_on_artifacts 查询
- 可观测性与 token 统计横切所有模块，放最后接入避免反复改动

## 约束

1. 每步大小合适，可独立验证
2. 步骤依赖关系正确
3. 严格符合 agent-runtime-refactor-design 设计文档目标，不缩水不遗漏
4. 每步遵循 TDD 流程（写测试→验证失败→实现→验证通过→commit）
5. 计划不含实现细节（无代码、无函数签名、无 SQL）
6. 每步有总结好的 commit message
7. 完备测试覆盖：UT + 集成测试 + 关键路径 E2E
8. 重构性质与切换时点：项目尚未发布，无生产数据需要保护，旧 `app/biz/agents/{runtime,service,scheduler}` 与 `execution_plans`/`task_outputs` 表可直接替换，无需数据迁移与双写过渡。按三个时点分阶段切换：
   - **切换点 A（步骤 5 完成后）**：API 路由层（`POST /projects/{id}/intents`、`POST /projects/{id}/agents/{type}/chat`、轻量调用）切到新 `AgentRuntime` 入口；删除旧 `app/biz/agents/routes.py` 的 dispatch/chat 路径
   - **切换点 B（步骤 6 完成后）**：`AgentScheduler` 与 `EventBus` 完全切到新 outbox 模型；删除旧 `app/biz/agents/scheduler.py`；新 `agent_runs`/`tasks`/`artifacts` 表成为唯一权威
   - **切换点 C（步骤 9 完成后）**：DROP `execution_plans`/`task_outputs` 表（迁移脚本 `DROP TABLE`）+ 删除 `app/biz/plans/` 与 `app/biz/outputs/` 模块；删除旧 `app/biz/agents/service.py` 等残留
   - 每个切换点单独 commit（如 `refactor: dispatch 路径切到新 AgentRuntime`），不与功能 commit 混杂
   - 切换点 C 前确认 `make check` 全部通过 + E2E-1 完整工作流在 staging 跑通

### - [ ] 步骤 7：失败恢复与生命周期

**增量**：`app/biz/agents/runtime.py` 增加失败处理与生命周期管理：模型调用重试策略（`tenacity` 区分错误类型：网络错误/HTTP 5xx/429 重试 3 次指数退避 1s/2s/4s + jitter；HTTP 400/401/403 不重试直接 failed；context_length_exceeded 上抛 ContextTooLongError 由调用方裁剪 history 重试一次）。Retry-After header 解析（delta-seconds 与 HTTP-date 两格式）。模型供方限流透传：5 分钟内同 user+model 触发供方 429 ≥3 次 → AdapterFactory 自动降级并发上限（5→2）持续 10 分钟，降级期间 dispatch 直接 ConcurrencyLimitError。`reconcile_on_startup()` 在 lifespan startup 执行：扫 `agent_runs WHERE status='running' AND started_at < process_start_time` → 标记 interrupted + cancel_reason=crash_recovery + 关联 task 回 pending + 通知 owner。`stop(run_id)`：agent_run→cancelled + 找到 in-flight asyncio.Task 调 `task.cancel()` + dispatch 捕获 CancelledError → task→paused（不清理 worktree）+ SDK 协作式取消最多等 30s。`resume(run_id)`：task→running + 新建 agent_run 复用 worktree。`cancel(task_id)`：task→cancelled + archive worktree + 关联 artifact→rejected。超时定时任务每 60s 扫描：dispatch 总时长 30min → agent_run→cancelled + task→timeout；模型调用 5min → ModelCallTimeoutError；Skill 30s → Skill handler TimeoutError。优雅停机 lifespan shutdown：`_shutting_down=True` + dispatch/chat/lightweight 端点返 503+Retry-After:60 + 等 in-flight run 最长 120s（每 5s 轮询）+ 超时强制 interrupted+system_shutdown + 关闭 AdapterFactory + 关闭 MCPManager + 关闭 DB 池。跨子系统事务补偿矩阵（步骤 1-16 失败点 + 补偿动作）；Outbox 模式实现（artifact INSERT + outbox_events INSERT 同事务，独立 task 推送到 EventBus）。
