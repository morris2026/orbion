# Orbion Git Worktree 模型 实施计划

## 策略

**数据模型 + GitCommandService 基础先行，逐层向上构建 WorktreeService**：先建 worktrees 表 + 封装 git 命令，再实现 WorktreeService 核心（create_or_reuse/archive/get/list + 6 状态映射），然后是三方合并、合并流程与冲突重生成，接着是 delete_by_owner + 权限，最后是事件、API、前端。

选择理由：

- GitCommandService 是 WorktreeService 的硬依赖，必须先封装 git 命令
- 全局 git 锁是所有 git 操作的前置，与 GitCommandService 同步实现
- WorktreeService 核心（create_or_reuse）依赖 GitCommandService + worktrees 表
- 三方合并是文件保存 API 的独立功能，可与 WorktreeService 核心并行
- 合并流程依赖 WorktreeService + 全局锁
- delete_by_owner 依赖 WorktreeService 核心 + archive
- 前端依赖后端 API 全部就绪

## 约束

1. 每步大小合适，可独立验证
2. 步骤依赖关系正确
3. 严格符合 git-worktree-model 设计文档目标，不缩水不遗漏
4. 每步遵循 TDD 流程（写测试→验证失败→实现→验证通过→commit）
5. 计划不含实现细节（无代码、无函数签名、无 SQL）
6. 每步有总结好的 commit message
7. 完备测试覆盖：UT + 集成测试

### - [x] 步骤 3：三方合并

### - [x] 步骤 4：合并流程 + 冲突重生成

**增量**：WorktreeService 新增 `merge(task_id, target_branch="main")` 方法：加全局 git 锁 → 锁内 double-check → `git merge task/{task_id}` 到 main worktree → 成功 archive / 冲突调 `regenerate()`；`regenerate(task_id)` 方法：`conflict_regen_count` 递增 → 超过 `max_conflict_regenerations`（构造函数注入，默认 3）抛 `WorktreeConflictError` + 清理旧 worktree → archive 当前 worktree → 基于最新 main 重建（DB INSERT 失败回滚 git 操作）；冲突时 worktree 标记 conflicting（短暂），自动重生成后回 active；超限后旧 worktree 已 archived，重新 dispatch 走首次创建分支。worktrees 表加 `conflict_regen_count` 列（MVP 暂存，agent-runtime 引入 tasks 表后迁移到 tasks 表）。新增 `WorktreeConflictError` / `WorktreeNotFoundError` / `GitOperationError` 异常。GitCommandService._run 补 `FileNotFoundError` 处理（main worktree 不存在时返回 returncode=127）。

**依赖**：步骤 2（WorktreeService 核心）+ 步骤 1（GitCommandService merge）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-4.1–GW-4.4

### - [x] 步骤 5：delete_by_owner + 权限

**增量**：WorktreeService 新增 `delete_by_owner(worktree_id, user_id)` 方法（AgentRuntime.cancel 的 worktree 层入口——AgentRuntime.cancel 负责 task 状态→cancelled + artifact→rejected，内部调用本方法完成 archive）；校验 worktree 存在 → main worktree 不可删（`worktree_type='main'` 抛 PermissionError）→ owner 校验（`created_by != user_id` 抛 PermissionError）→ task 状态校验（completed/cancelled 抛 `TaskStateError`，通过 TaskContext.task_status 获取）→ 调用 `archive()` 清理 worktree。TaskContext 加 `task_status: str | None` 字段。新增 `TaskStateError` 异常。task.status → cancelled 不在本步骤范围（AgentRuntime.cancel 负责）。stop/resume/cancel 属于 AgentRuntime 层接口，本步骤只实现 worktree 层的 delete_by_owner。
