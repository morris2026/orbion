# Orbion Git Worktree 模型 实施计划

## 策略

**数据模型 + GitService 基础先行，逐层向上构建 WorktreeService**：先建 worktrees 表 + 封装 git 命令，再实现 WorktreeService 核心（create_or_reuse/archive/get/list + 6 状态映射），然后是三方合并、合并流程与冲突重生成，接着是 delete_by_owner + 权限，最后是事件、API、前端。

选择理由：

- GitService 是 WorktreeService 的硬依赖，必须先封装 git 命令
- 全局 git 锁是所有 git 操作的前置，与 GitService 同步实现
- WorktreeService 核心（create_or_reuse）依赖 GitService + worktrees 表
- 三方合并是文件保存 API 的独立功能，可与 WorktreeService 核心并行
- 合并流程依赖 WorktreeService + 全局锁
- delete_by_owner 依赖 WorktreeService 核心 + archive
- 前端依赖后端 API 全部就绪

## 约束

1. 每步大小合适，可独立验证
2. 步骤依赖关系正确
3. 严格符合 git-worktree-model 设计文档目标，不缩水不遗漏
4. 每步遵循 TDD 流程（写测试→验证失败→实现→验证通过→commit）
5. 计划不含实现细节（无代码、无函数签名、无 SQL）
6. 每步有总结好的 commit message
7. 完备测试覆盖：UT + 集成测试

共 7 步：

| 步骤 | 核心增量 | 对应验证点 |

### - [ ] 步骤 2：全局 git 锁 + WorktreeService 核心

**增量**：新建 app/biz/git/git_lock.py 实现进程级全局 asyncio.Lock（_global_git_lock）；所有 git 操作通过 with _global_git_lock 串行化；新建 app/biz/worktree/worktree_service.py 实现 create_or_reuse(task_id)（首次创建 git worktree add + branch，重做复用现有）/ archive(task_id)（worktree remove + branch delete + 表状态 archived）/ get(worktree_id) / list_by_project(project_id)；worktrees 表 status 字段 active/conflicting/archived 三态流转；task 6 状态与 worktree 状态映射（pending 首次无/cancelled 后保留、running active、paused 保留、completed active、timeout 保留、cancelled archived）

**依赖**：步骤 1（GitService + worktrees 表）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-2.1–GW-2.8

### - [ ] 步骤 3：三方合并

**增量**：GitService 新增 merge_file(mine, original, theirs) 方法封装 git merge-file --diff3；文件保存 API 接收 expected_mtime + original_content + current_content 参数；后端检测当前文件 mtime vs expected_mtime，相同直接保存，不同执行三方合并；合并成功（返回 0）原子写入（tmp + rename）；合并失败（返回非 0）返回 409 Conflict + merged_content + conflict_markers；前端用 Monaco Diff Editor 展示冲突；30 分钟未保存提示重新加载

**依赖**：步骤 1（GitService 基础）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-3.1–GW-3.5

**commit message**：`feat: 文件保存三方合并与冲突处理`

### - [ ] 步骤 4：合并流程 + 冲突重生成

**增量**：WorktreeService 新增 merge(task_id, target_branch="main") 方法：加全局 git 锁 → git merge task/{task_id} → 冲突时调 regenerate()；regenerate(task_id) 方法：tasks.conflict_regen_count 递增 → 超过 MAX_CONFLICT_REGENERATIONS（默认 3，runtime.json 可配）抛 WorktreeConflictError → archive 当前 worktree → create_or_reuse 基于最新 main 重建 → 返回新 Worktree；冲突时 task 保持 completed，worktree 标记 conflicting（短暂），自动重生成后 worktree 回 active；冲突超限 task 回 pending；revision 场景 revision_notes 通过事件 payload 传递

**依赖**：步骤 2（WorktreeService 核心）+ 步骤 1（GitService merge）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-4.1–GW-4.5

**commit message**：`feat: 合并流程与冲突自动重生成`

---

### - [ ] 步骤 5：delete_by_owner + 权限

**增量**：WorktreeService 新增 delete_by_owner(worktree_id, user_id) 方法（AgentRuntime.cancel 的 worktree 层入口——AgentRuntime.cancel 负责 task 状态→cancelled + artifact→rejected，内部调用本方法完成 archive）；校验 user_id 是 worktree owner + task 状态为 pending/running/paused/timeout（completed/cancelled 不允许）→ 调用 archive()；非 owner 抛 PermissionError；task 状态不允许抛 TaskStateError；main worktree 不可删（强制校验 worktree_type='task'）；跨用户审查场景其他用户切换到 task worktree 只读。注意：stop(run_id) / resume(run_id) / cancel(task_id) 属于 AgentRuntime 层接口（详见 agent-runtime §8.2），本步骤只实现 worktree 层的 delete_by_owner + archive 支持

**依赖**：步骤 2（WorktreeService 核心 + archive）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-5.1–GW-5.6

**commit message**：`feat: delete_by_owner 与权限校验`

### - [ ] 步骤 6：事件发布 + API 端点

**增量**：新建 4 个事件类型 WorktreeCreated/WorktreeMerged/WorktreeConflictDetected/WorktreeArchived，通过 EventBus 发布（payload 含 worktree_id/project_id/type/branch_name/target_branch/commit_sha/conflict_files 等）；新建 app/api/worktrees/routes.py 实现 GET /projects/{project_id}/worktrees（列表）/ GET /worktrees/{id}（详情）/ DELETE /worktrees/{id}（cancel，仅 owner，对应 delete_by_owner）/ POST /worktrees/{id}/merge（合并）；文件操作 API 增加 worktree 上下文：GET/POST/DELETE /projects/{project_id}/worktrees/{id}/files + GET /worktrees/{id}/files/tree + POST /worktrees/{id}/files/analyze。注意：stop/resume 的 HTTP 端点属于 AgentRuntime 层（agent-runtime §12 定义），不在本步骤范围

**依赖**：步骤 2（WorktreeService 核心）+ 步骤 4（merge）+ 步骤 5（delete_by_owner）

**测试设计**：见 [1.10-mvp-git-worktree-model-test-design](1.10-mvp-git-worktree-model-test-design.md) GW-6.1–GW-6.6

**commit message**：`feat: worktree 事件发布与 API 端点`

**增量**：web/src/components/workspace/FileTab/ 顶部新增 worktree 选择器下拉（显示 main + 所有 task worktree + 状态标签）；切换 worktree 重新加载文件树/编辑器/Source Control；main worktree 切换时编辑器只读模式（Monaco readOnly=true）；task worktree 切换时其他用户只读（通过 API 返回的 owner 字段判断）；文件保存 API 传 expected_mtime + original_content；409 响应弹 Monaco Diff Editor 冲突对话框（对比并合并/覆盖/取消三选项）；无"新建 worktree"按钮

**依赖**：步骤 6（API 端点全部就绪）+ 步骤 3（三方合并 409 响应）
