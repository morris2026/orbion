# Orbion Git Worktree 模型 测试设计

本文档为 [1.10-mvp-git-worktree-model-impl-plan](1.10-mvp-git-worktree-model-impl-plan.md) 的配套测试设计，按实施计划步骤顺序组织。每个步骤的验证标准在此展开为具体用例划分、主体流程和检查项。

## 用例格式

每个用例包含：

- **用例名**：概括测试意图
- **类型**：UT / API / 集成（本计划为后端模块 + 前端集成）
- **主体流程**：操作序列
- **检查项**：断言要点

---

## 步骤 1：数据模型 + GitService 基础

### GW-1.1 worktrees 表迁移成功

- **类型**：UT
- **主体流程**：执行 Alembic 迁移创建 worktrees 表
- **检查项**：worktrees 表存在；字段 id/project_id/repo_name/worktree_type/branch_name/path/status/created_by/task_id/created_at/updated_at 齐全；INDEX(project_id, worktree_type) 和 INDEX(task_id) 生效

### GW-1.2 GitService worktree add 封装

- **类型**：UT
- **主体流程**：在 tmp_path 下初始化 bare 仓库；调用 git_service.worktree_add(bare_repo_path, "worktrees/task_001", "task/001", "main")
- **检查项**：worktrees/task_001 目录存在；git worktree list 含 task/001 分支；返回 WorktreeResult(success=True)

### GW-1.3 GitService worktree remove 封装

- **类型**：UT
- **主体流程**：已存在 task_001 worktree；调用 git_service.worktree_remove(path)
- **检查项**：worktrees/task_001 目录不存在；git worktree list 不再含该条目；返回 success=True

### GW-1.4 GitService branch 创建与删除

- **类型**：UT
- **主体流程**：调用 git_service.branch_create(repo, "task/001", "main")；再调用 git_service.branch_delete(repo, "task/001")
- **检查项**：创建后 git branch --list 含 task/001；删除后不含

### GW-1.5 GitService merge 成功

- **类型**：UT
- **主体流程**：main 分支有 1 个 commit，task/001 分支有 1 个不同文件 commit；调用 git_service.merge(target_wt, "task/001")
- **检查项**：返回 MergeResult(success=True, has_conflicts=False)；target_wt 工作目录含两边的修改

### GW-1.6 GitService merge 冲突

- **类型**：UT
- **主体流程**：main 和 task/001 修改同文件同位置；调用 git_service.merge
- **检查项**：返回 MergeResult(success=False, has_conflicts=True, conflict_files=[...])；merge 自动 abort（target_wt 回到合并前状态）

## 步骤 2：全局 git 锁 + WorktreeService 核心

### GW-2.1 全局锁串行化 git 操作

- **类型**：UT
- **主体流程**：并发调用 5 次 git_service.commit（不同 worktree），测量耗时
- **检查项**：5 次操作串行执行（总耗时 ≈ 单次 × 5）；无 .git/index.lock 冲突错误

### GW-2.2 create_or_reuse 首次创建

- **类型**：UT
- **主体流程**：task_001 无 worktree；调用 worktree_service.create_or_reuse(task_id="task_001")
- **检查项**：worktrees 表新增记录 worktree_type='task' status='active' task_id='task_001'；文件系统 worktrees/task_001 目录存在；branch task/001 已创建

### GW-2.3 create_or_reuse 重做复用

- **类型**：UT
- **主体流程**：task_001 已有 active worktree（含之前产出）；再次调用 create_or_reuse(task_id="task_001")
- **检查项**：不新建 worktree（文件系统目录不变）；返回的 Worktree 与已有记录一致；worktrees 表无新增

### GW-2.4 archive 清理 worktree

- **类型**：UT
- **主体流程**：task_001 有 active worktree；调用 archive(task_id="task_001")
- **检查项**：文件系统 worktrees/task_001 目录不存在；git branch task/001 已删除；worktrees 表 status='archived'

### GW-2.5 get 查询 worktree

- **类型**：UT
- **主体流程**：插入 1 条 worktree 记录；调用 get(worktree_id)
- **检查项**：返回 Worktree 对象含所有字段；不存在的 id 返回 None

### GW-2.6 list_by_project 列出项目 worktree

- **类型**：UT
- **主体流程**：项目 p1 有 main + 2 个 task worktree；调用 list_by_project(project_id="p1")
- **检查项**：返回 3 条记录；含 main worktree（worktree_type='main'）；task worktree 按 created_at 倒序

### GW-2.7 跨项目并发 git 锁串行化但不死锁

- **类型**：集成
- **主体流程**：项目 p1 和 p2 各自有 bare 仓库；并发调用 p1 的 commit 和 p2 的 merge（共 4 个并发任务，每项目 2 个）；等待全部完成
- **检查项**：4 个任务全部成功完成（无死锁）；每个任务的结果与单独执行一致；git 操作串行化（通过日志或 metric 验证全局锁被多次 acquire/release）；无 .git/index.lock 冲突错误

### GW-2.8 task 6 状态与 worktree 状态映射

- **类型**：UT
- **主体流程**：模拟 task 从 pending → running → paused → running → completed → cancelled 全流程，每步调用 create_or_reuse / stop / resume / archive
- **检查项**：pending（首次）worktree 不存在；running worktree active；paused worktree 保留；completed worktree active；cancelled worktree archived

## 步骤 3：三方合并

### GW-3.1 mtime 相同直接保存

- **类型**：API
- **主体流程**：用户 A 打开文件（前端记录 mtime=T1）；A 保存（expected_mtime=T1）；后端检测当前文件 mtime 仍为 T1
- **检查项**：直接保存成功；响应 200 + 新 mtime；未触发三方合并

### GW-3.2 mtime 不同自动合并成功

- **类型**：API
- **主体流程**：A 打开文件（mtime=T1）；B 保存修改同一文件不同位置（mtime 变为 T2）；A 保存（expected_mtime=T1，original_content=T1 时的内容，current_content=A 的修改）
- **检查项**：后端检测 mtime 变化；执行 git merge-file；返回 200 + 合并后的内容；A 的修改与 B 的修改都保留（不同位置）

### GW-3.3 同位置修改返回 409

- **类型**：API
- **主体流程**：A 和 B 修改同一文件同一位置；A 后保存（expected_mtime 已过期）
- **检查项**：返回 409；响应含 merged_content（带冲突标记）；响应含 conflict_markers（冲突位置列表）

### GW-3.4 冲突对话框三选项

- **类型**：集成
- **主体流程**：前端收到 409 响应；用户选择"对比并合并"
- **检查项**：Monaco Diff Editor 展示冲突；用户手动解决后保存成功；选择"覆盖"时用 A 的版本覆盖磁盘；选择"取消"时编辑器内容保留

### GW-3.5 30 分钟未保存提示重新加载

- **类型**：集成
- **主体流程**：用户打开文件后 31 分钟未保存；尝试保存
- **检查项**：前端提示"文件已过期，建议重新加载"；用户确认后重新加载文件（放弃当前编辑或选择合并）

## 步骤 4：合并流程 + 冲突重生成

### GW-4.1 merge 成功流转

- **类型**：UT
- **主体流程**：task_001 产出审批通过（task completed）；调用 worktree_service.merge(task_id="task_001", target_branch="main")
- **检查项**：main 分支含 task/001 的 commit；worktree_service.archive 被调用；worktrees 表 status='archived'

### GW-4.2 merge 冲突自动重生成

- **类型**：UT
- **主体流程**：main 推进后 task_001 合并冲突；调用 merge
- **检查项**：task 保持 completed，worktree 标记 conflicting（短暂）；tasks.conflict_regen_count 递增到 1；regenerate() 被调用；新 worktree 基于最新 main 创建；新 agent_run 被触发（通过 mock 验证）

### GW-4.3 重生成次数超限 task 回 pending

- **类型**：UT
- **主体流程**：task_001 已重生成 3 次（conflict_regen_count=3）；第 4 次冲突
- **检查项**：抛 WorktreeConflictError；task 状态转为 pending；前端提示"冲突次数过多，需人工介入"

### GW-4.4 MAX_CONFLICT_REGENERATIONS 可配

- **类型**：UT
- **主体流程**：runtime.json 设置 max_conflict_regenerations=5；task 重生成 5 次后第 6 次冲突
- **检查项**：前 5 次正常重生成；第 6 次抛 WorktreeConflictError

### GW-4.5 重做场景 revision_notes 传递

- **类型**：UT
- **主体流程**：task_001 的 artifact 被审批拒绝，task 回到 pending（含审批者修改意见）；冲突重生成触发新 dispatch
- **检查项**：新 dispatch 的事件 payload 含 revision_notes 字段；revision_notes.source='approver'；notes 列表含原审批者意见

## 步骤 5：delete_by_owner + 权限

> 注意：stop(run_id) / resume(run_id) / cancel(task_id) 属于 AgentRuntime 层接口（详见 agent-runtime §8.2），其测试在 agent-runtime 测试设计中覆盖。本步骤测试 WorktreeService 层的 delete_by_owner（AgentRuntime.cancel 内部调用本方法完成 worktree archive）。

### GW-5.1 delete_by_owner 从 paused 状态放弃 task

- **类型**：UT
- **主体流程**：task_001 处于 paused，有 active worktree；owner 调用 delete_by_owner(worktree_id, user_id=owner)
- **检查项**：worktree 被 archive（文件清理，表 status='archived'）；task.status → cancelled

### GW-5.2 delete_by_owner 从 running 状态放弃 task

- **类型**：UT
- **主体流程**：task_001 处于 running，有 active worktree；owner 调用 delete_by_owner(worktree_id, user_id=owner)
- **检查项**：worktree 被 archive；task.status → cancelled

### GW-5.3 delete_by_owner 从 pending/timeout 状态放弃 task

- **类型**：UT
- **主体流程**：分别对 pending（首次无 worktree）和 timeout（有保留 worktree）状态的 task 调用 delete_by_owner
- **检查项**：timeout → worktree 被 archive，task → cancelled；pending（无 worktree）→ task → cancelled，archive 为 no-op（无文件可删，不抛错）

### GW-5.4 delete_by_owner 非 owner 抛 PermissionError

- **类型**：UT
- **主体流程**：user_b 不是 task_001 的 owner；调用 delete_by_owner(worktree_id, user_id=user_b)
- **检查项**：抛 PermissionError；worktree 未被删除；task 状态不变

### GW-5.5 delete_by_owner 对 completed/cancelled 状态抛 TaskStateError

- **类型**：UT
- **主体流程**：task_001 处于 completed（或 cancelled）；owner 调用 delete_by_owner(worktree_id, user_id=owner)
- **检查项**：抛 TaskStateError；worktree 未被删除；task 状态不变

### GW-5.6 main worktree 不可删

- **类型**：UT
- **主体流程**：调用 delete_by_owner(main_worktree_id, user_id=任意)
- **检查项**：抛 PermissionError 或 ValueError；main worktree 未被删除

## 步骤 6：事件发布 + API 端点

### GW-6.1 WorktreeCreated 事件发布

- **类型**：集成
- **主体流程**：调用 worktree_service.create_or_reuse(task_id="task_001")
- **检查项**：EventBus 发布 WorktreeCreated 事件；payload 含 worktree_id/project_id/type='task'/branch_name/created_by

### GW-6.2 WorktreeMerged 事件发布

- **类型**：集成
- **主体流程**：merge 成功
- **检查项**：EventBus 发布 WorktreeMerged 事件；payload 含 worktree_id/target_branch/commit_sha

### GW-6.3 WorktreeConflictDetected 事件发布

- **类型**：集成
- **主体流程**：merge 冲突
- **检查项**：EventBus 发布 WorktreeConflictDetected 事件；payload 含 worktree_id/target_branch/conflict_files

### GW-6.4 GET /worktrees 列表

- **类型**：API
- **主体流程**：项目 p1 有 main + 2 task worktree；GET /projects/p1/worktrees
- **检查项**：响应 200；返回 3 条 worktree 记录；每条含 id/type/branch_name/status/task_id

### GW-6.5 DELETE /worktrees/{id} cancel

- **类型**：API
- **主体流程**：user_a 是 owner；DELETE /projects/p1/worktrees/{id}（认证 user_a）
- **检查项**：响应 204；worktree 被 archive；task status → cancelled

### GW-6.6 文件操作 API 含 worktree 上下文

- **类型**：API
- **主体流程**：GET /projects/p1/worktrees/{id}/files?path=README.md
- **检查项**：响应 200 含 content/mtime/size；读取的是指定 worktree 的文件（不是 main）；POST/DELETE 同样按 worktree 上下文操作

## 步骤 7：前端 worktree 选择器 + file tab 集成

### GW-7.1 worktree 选择器下拉

- **类型**：集成
- **主体流程**：项目有 main + task_001（执行中）+ task_002（待审批）；打开 file tab
- **检查项**：下拉显示 3 个选项；每个选项含状态标签（main / task_001 执行中 / task_002 待审批）；默认选中 main

### GW-7.2 切换 worktree 重新加载

- **类型**：集成
- **主体流程**：从 main 切换到 task_001
- **检查项**：文件树重新加载（显示 task_001 worktree 的文件）；编辑器内容清空（等待用户选择文件）；Source Control 面板显示 task_001 的 staged/unstaged/commits

### GW-7.3 main worktree 只读

- **类型**：集成
- **主体流程**：切换到 main worktree；打开文件尝试编辑
- **检查项**：Monaco 编辑器 readOnly=true；编辑器顶部显示"只读"标识；保存按钮禁用

### GW-7.4 无"新建 worktree"按钮

- **类型**：集成
- **主体流程**：检查 file tab 顶部工具栏
- **检查项**：仅有 worktree 选择器下拉；无"新建 worktree"按钮（用户不能主动创建 worktree，只能由系统在 dispatch 时创建）
