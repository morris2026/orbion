# Orbion Agent Runtime 重构 测试设计

本文档为 [1.11-mvp-agent-runtime-refactor-impl-plan](1.11-mvp-agent-runtime-refactor-impl-plan.md) 的配套测试设计，按实施计划步骤顺序组织。每个步骤的验证标准在此展开为具体用例划分、主体流程和检查项。

## 用例格式

每个用例包含：

- **用例名**：概括测试意图
- **类型**：UT / 集成 / E2E（本计划为后端模块为主，前端集成按需）
- **主体流程**：操作序列
- **检查项**：断言要点

### 命令封装类用例原则

`AdapterFactory`、`SkillPermissionChecker` 这类封装层的用例必须同时覆盖 happy path 和 error path：

- **happy path**：常规场景，验证返回值 + 副作用（DB 状态/缓存命中）
- **error path**：异常场景（密钥缺失、缓存失效、限流、超时等），验证返回结构化错误而非抛异常

Why：封装层的核心契约是"不抛异常、返回结构化结果或可重试错误"——happy path 无法验证这个契约，只有 error path 用例才能暴露异常泄漏。error path 用例编号在原编号后加字母后缀（如 AR-4.3a）。

### 模型 mock 策略

所有用例的模型调用必须 mock（避免烧真实 token + 测试不稳定）：

- 实现 `FakeAgentRuntimeAdapter`，按预设脚本返回 `AgentEvent` 序列（dispatch/chat 场景，覆盖 text_delta/tool_call/tool_result/permission_request/done/error）与 `StreamChunk` 序列（lightweight_call 场景，覆盖 text/done/error）
- 集成测试通过依赖注入替换 `AdapterFactory`，注入 fake adapter
- E2E 测试使用 `scripts/start-e2e-server.py` 注入的 `FakeAgentRuntimeAdapter` 模式（确定性结构化 JSON 响应，按 task 关键词路由）

### 测试位置约定

| 类型 | 目录 | 文件命名 | 备注 |
| ---- | ---- | -------- | ---- |

## 步骤 1：数据模型基础

### AR-1.1 user_models/artifacts/tasks/agent_runs 等新表迁移成功

- **类型**：UT
- **主体流程**：执行 `migrations/` 下所有新迁移脚本，验证表结构
- **检查项**：user_models 表（api_key_enc BYTEA / api_key_hash / 唯一键 (user_id, model_id)）；artifacts 表（based_on_artifacts JSONB / GIN 索引 / status_changed_at / last_reminded_at）；tasks 表（6 状态 CHECK / based_on_artifacts / based_on_tasks / revision_count / conflict_regen_count）；agent_runs 表（5 状态 / cancel_reason / trace_id / UNIQUE(event_id, agent_type) / 部分唯一索引 agent_runs_task_running_idx ON (task_id) WHERE status='running'）；model_usage_details/daily/archive 三表；skill_calls 审计表；outbox_events + dead_letter_events
- **测试位置**：合入 `tests/integration/test_postgres_infrastructure.py::test_postgres_migration_creates_tables_and_indexes`

### AR-1.2 artifacts.based_on_artifacts GIN 索引生效

- **类型**：UT
- **主体流程**：插入 3 条 artifact（A based_on []; B based_on [A]; C based_on [B]）；查询 `WHERE based_on_artifacts @> '[{"artifact_id": "A"}]'`
- **检查项**：返回 B（直接依赖 A）；不返回 C（间接依赖靠应用层递归查询）；EXPLAIN 显示走 GIN 索引（无 seq scan）

### AR-1.3 agent_runs 部分唯一索引阻止同 task 重入

- **类型**：UT
- **主体流程**：插入 task_id=T1 status=running 的 agent_run；再插入同 task_id 第二条 running
- **检查项**：第二条 INSERT 抛 UniqueViolationError；插入 status=completed 不冲突；插入其他 task_id 的 running 不冲突

### AR-1.4 agent_runs (event_id, agent_type) 幂等键生效

- **类型**：UT
- **主体流程**：插入 (event_id=E1, agent_type=implementer)；再插入同 event_id + 同 agent_type
- **检查项**：抛 UniqueViolationError；同 event_id + 不同 agent_type 不冲突

### AR-1.5 model_usage_daily 联合唯一键 (user_id, model_id, agent_type, date) 生效

- **类型**：UT
- **主体流程**：UPSERT 同四元组两次
- **检查项**：第二次走 ON CONFLICT DO UPDATE（call_count 累加）；不同四元组各自独立

### AR-1.6 model_usage_archive 联合唯一键 (user_id, date) 生效

- **类型**：UT
- **主体流程**：INSERT 同 (user_id, date) 两次
- **检查项**：第二次按 ON CONFLICT DO NOTHING 跳过；不同 (user_id, date) 各自独立

### AR-1.7 event_log.payload_ref_url 字段 + L2 引用约定

- **类型**：UT
- **主体流程**：执行迁移；插入 event_log 行 payload={"instruction": "..."}（< 4KB，L1 内联）；插入 payload_ref_url="s3://orbion-events/{event_id}/revision_notes"（L2 引用）
- **检查项**：payload JSONB 字段存在；payload_ref_url VARCHAR NULL 字段存在；两者不同时非空（L1 内联时 ref_url 为 NULL，L2 引用时 payload 含 ref_url 占位）

## 步骤 2：用户模型配置 + 加密

### AR-2.1 AES-GCM 加密-解密往返一致性

- **类型**：UT
- **主体流程**：设置 `ORBION_ENCRYPTION_KEY` 环境变量；加密 "sk-test-key-123" → 解密
- **检查项**：解密结果与原文严格相等；密文每次不同（nonce 随机）；密文格式 `nonce(12B) || ciphertext || tag(16B)`

### AR-2.2 ORBION_ENCRYPTION_KEY 缺失时服务拒绝启动

- **类型**：UT
- **主体流程**：unset 环境变量；调用 `app.main` 启动 lifespan
- **检查项**：抛 RuntimeError 或 AssertionError；服务未绑定端口

### AR-2.3 UserModel CRUD + api_key_hash 变更检测

- **类型**：集成
- **主体流程**：POST 创建 UserModel（api_key="sk-abc"）→ GET 列表（不返回 api_key，返回 masked "sk-***bc"）→ PUT 更新（不传 api_key）→ PUT 更新（传新 api_key="sk-xyz"）
- **检查项**：第一次 PUT 后 api_key_enc 不变（hash 未变）；第二次 PUT 后 api_key_enc 变化（hash 变化触发重新加密）；GET 永不返回明文 api_key

### AR-2.4 AgentModelMapping 四级解析顺序

- **类型**：集成
- **主体流程**：用户级配置 analyst→GLM-4；项目级 override 配置 analyst→Claude；用户只有 1 个 UserModel（未配 AgentModelMapping）
- **检查项**：有项目级 override → 返回项目级（Claude）；无 override 时返回用户级（GLM-4）；用户级也未配 + 单 UserModel → 返回智能默认（该唯一 UserModel）；都没配置 → 抛 ModelNotConfiguredError

### AR-2.5 单 UserModel 智能默认

- **类型**：集成
- **主体流程**：用户只有 1 个 UserModel（未配 AgentModelMapping）；解析各 agent_type 的模型
- **检查项**：6 个 agent_type 都返回该唯一 UserModel.model_id

### AR-2.6 多 UserModel 未配置时引导

- **类型**：集成
- **主体流程**：用户有 2 个 UserModel，未配 AgentModelMapping；解析 analyst 的模型
- **检查项**：抛 ModelNotConfiguredError，HTTP 409；错误响应含 `configure_url: /users/me/agent-models`

### AR-2.7 agent_models.enc 加密文件读写

- **类型**：UT
- **主体流程**：写 agent_models.enc → 读 → 修改 → 再读
- **检查项**：文件内容是加密字节（非明文 JSON）；读出的 dict 与写入一致；HKDF info=b"orbion-agent-models-key" 与 credentials 隔离（用错 key 解密失败）

### AR-2.8 UserModel 被 AgentModelMapping 引用时禁止删除

- **类型**：集成
- **主体流程**：创建 UserModel "我的 GLM-4" → 配置 analyst→"我的 GLM-4" → DELETE UserModel
- **检查项**：DELETE 返回 409；错误消息明确"被 analyst 引用"；解除引用后 DELETE 成功

## 步骤 3：AgentRuntimeAdapter 基础设施

### AR-3.1 BaseAdapter.complete() 内部复用 stream()

- **类型**：UT
- **主体流程**：定义 FakeAgentRuntimeAdapter 继承 BaseAdapter，stream() 按 [text/text/done] 脚本 yield StreamChunk；调 complete()
- **检查项**：complete() 返回 ModelOutput.content = 两个 text chunk delta 拼接；ModelOutput.usage = done chunk 的 usage

### AR-3.2 流式 done StreamChunk 携带完整 usage

- **类型**：UT
- **主体流程**：FakeAgentRuntimeAdapter stream() yield [text StreamChunk] × 3 → [done StreamChunk with usage]
- **检查项**：done chunk 之前的 chunk.usage 是 None；done chunk.usage 完整；BaseAdapter.complete() 提取 done chunk 的 usage 写入 ModelOutput
- **验证目标**：§6.1 token 统计在流式下如何收口 + §6.1.2 StreamChunk 定义

### AR-3.3 供方响应无 usage 时本地估算兜底

- **类型**：UT（error path）

- **检查项**：UsageInfo.estimated=True；input_tokens + output_tokens 用 tiktoken 估算（非零）；model_name 正确
- **验证目标**：§6.7.2 兜底策略

### AR-3.4 error chunk 触发 ModelCallError

- **类型**：UT

- **验证目标**：§6.7.3 invalidate() API

- **类型**：UT

### AR-4.5 并发单实例化

- **类型**：UT

- **检查项**：5 次返回相同实例（仅一次 _build 调用）；锁未死锁

## 端到端测试（E2E）

E2E 用例位于 `web/e2e/agent-runtime.spec.ts`，复用 `self-build.spec.ts` 的 ADMIN 凭据与 `loginAs` / `authHeaders` / `createProject` 工具函数。每用例创建唯一项目（名称带 timestamp）避免数据冲突，afterAll 调 `DELETE /projects/{id}` 清理。

E2E 通过 `scripts/start-e2e-server.py` 注入的 `FakeAgentRuntimeAdapter` 获得确定性响应（按 task 关键词返回结构化 JSON），不烧真实 token。前端通过真实浏览器操作 UI 触发意图命令与审批。

### E2E-1a 需求→系统→子系统工作流

- **类型**：E2E
- **主体流程**：登录 admin → 创建项目 + 仓库 → 讨论区发 `/analyze 重点总结性能讨论` → 等 Agent 完成 → 审批 requirement artifact（approve）→ 发 `/design req-001 用微服务架构` → 审批 system artifact → 发 `/decompose sys-001 按业务域拆分子系统` → 发 `/design subsystem-001 认证子系统详细设计` → 审批 subsystem artifact
- **检查项**：每步 agent_run 状态流转正确（running→completed）；artifact 状态机（draft→proposed→approved）；task 6 状态生命周期；worktree 在 task 完成后保留；事件顺序按 thread_id 串行；基于上游 artifact 的下游 task 依赖校验生效
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-1b 子系统→模块→实施→合并工作流

- **类型**：E2E
- **主体流程**：承接 E2E-1a 的 subsystem artifact → `/decompose sub-001 拆分实施任务` → `/design module-001` → 审批 module artifact → `/decompose mod-001` → `/implement`（无参数执行所有 pending development task）→ 审查 worktree diff → approve 触发 merge → 验证 main 分支含新代码
- **检查项**：development task 不产 artifact（output_artifact_id IS NULL）；worktree diff 审批路径；approve 触发 WorktreeService.merge 到 main；merge 后 main 含产出代码；下游 task 在上游 module artifact approved 后解锁
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-2 Critic 拒绝与重做循环

- **类型**：E2E
- **主体流程**：项目配置 critic_enabled=true + FakeAgentRuntimeAdapter 改造为 critic 首次返回 verdict=reject → `/analyze` → critic 拒绝（revision_count=1）→ task 回 pending → 重新 dispatch → critic 再次拒绝（revision_count=2，达 MAX_CRITIC_REVISIONS）→ artifact 保持 proposed + 前端高亮"⚠️ critic 已拒绝 2 次" → 用户手动 approve（override critic）
- **检查项**：revision_count 递增；达上限后不再自动回 pending；前端高亮显示；用户 override 后 artifact→approved；解锁下游
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-3 变更传播 owner 确认

- **类型**：E2E
- **主体流程**：完成 `/analyze` + `/design req-001` 后 requirement + system 均 approved → 修改 requirement artifact（重新 dispatch /analyze 产出新版本）→ 系统识别 system based_on [requirement] → 流程 tab 显示"待确认变更"卡片 → owner 点确认 → 系统自动发布 DesignRequested 事件 → architect 重新设计 system
- **检查项**：变更传播识别下游（based_on_artifacts GIN 查询）；待确认变更卡片置顶显示；owner 确认后自动 dispatch；新 agent_run.metadata.source=auto_propagate；不强制用户接受变更
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-4 多用户并发 + worktree 隔离

- **类型**：E2E
- **主体流程**：注册 member2 + admin 将其加为项目成员 → 两人同时在不同浏览器 context 发 `/design subsystem-001` 与 `/design subsystem-002`（不同 task）→ 等两个 dispatch 完成
- **检查项**：两个 worktree 路径独立（task/{task_id_1} vs task/{task_id_2}）；两个 agent_run 并发创建不串行；file.write 各自只在 worktree 子树内；全局并发信号量未触发拒绝（≤20）
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-5 崩溃恢复（进程重启后 reconcile）

- **类型**：E2E
- **主体流程**：发 `/implement task-001` 起 long-running dispatch（FakeAgentRuntimeAdapter 模拟慢响应）→ dispatch 进行中重启 e2e-server 进程（docker restart 或 kill -9 + 自动重启）→ 等服务恢复 → 调 `GET /projects/{id}/runs/{run_id}`
- **检查项**：agent_run.status=interrupted；cancel_reason=crash_recovery；task.status 回到 pending；owner 收到 in_app 通知；用户可重新 dispatch（复用 worktree 半成品）
- **测试位置**：`web/e2e/agent-runtime.spec.ts`

### E2E-6 模型限流场景下的用户体验

- **类型**：E2E
- **主体流程**：FakeAgentRuntimeAdapter 改造为连续返回供方 HTTP 429（抛 ModelProviderRateLimitError + retry_after_seconds=30）→ 发 `/analyze` 三次 → 观察前端提示 → 第四次发 `/analyze` → 等 10 分钟降级窗口（用 fake clock 加速）→ 第五次发 `/analyze`
- **检查项**：前三次客户端收 HTTP 503 ModelProviderRateLimit + 前端显示"[模型名] 暂时限流，约 30 秒后可恢复"（供方 429 透传）；第四次触发自适应降级，客户端收 HTTP 429 ConcurrencyLimit + 前端显示"已临时降级（剩余 2 并发）"（平台级限流，与供方 503 语义不同）；降级窗口结束后第五次恢复正常 dispatch
- **测试位置**：`web/e2e/agent-runtime.spec.ts`
