# Orbion 文件系统工作目录 测试设计

本文档为 [1.8-mvp-file-layout-impl-plan](1.8-mvp-file-layout-impl-plan.md) 的配套测试设计，按实施计划步骤顺序组织。每个步骤的验证标准在此展开为具体用例划分、主体流程和检查项。

## 用例格式

每个用例包含：
- **用例名**：概括测试意图
- **类型**：UT / API / 集成（本计划为 MVP 局部重构，不涉及 E2E/Benchmark/构建验证）
- **主体流程**：操作序列
- **检查项**：断言要点

---

## 步骤 1：Settings 重构 — orbion_dir 替代双路径

### MVP-FL-1.1 Settings 默认 orbion_dir

- **类型**：UT
- **主体流程**：创建 Settings 实例（jwt_secret 通过环境变量注入）
- **检查项**：`orbion_dir` 为 `"./data"`；`repo_path` 和 `memory_base_path` 属性不存在（hasattr 返回 False）

### MVP-FL-1.2 ORBION_DIR 环境变量覆盖

- **类型**：UT
- **主体流程**：设置 `ORBION_DIR=/var/lib/orbion` 后创建 Settings 实例
- **检查项**：`orbion_dir` 为 `"/var/lib/orbion"`

### MVP-FL-1.3 派生路径计算 — 默认值

- **类型**：UT
- **主体流程**：默认 `orbion_dir = "./data"` 下调用各派生路径方法
- **检查项**：
  - `projects_dir` → `./data/projects`
  - `platform_memory_path` → `./data/memory.md`
  - `project_dir("p1")` → `./data/projects/p1`
  - `project_memory_path("p1")` → `./data/projects/p1/memory.md`
  - `agent_memory_path("p1", "summary")` → `./data/projects/p1/agents/summary/memory.md`
  - `project_repo_path("p1", "orbion")` → `./data/projects/p1/repo/orbion`
  - `thread_dir("p1", "t1")` → `./data/projects/p1/threads/t1`（接口预留，MVP 不调用）
  - `output_payload_path("p1", "o1")` → `./data/projects/p1/outputs/o1.json`（接口预留，MVP 不调用）

### MVP-FL-1.4 派生路径计算 — 绝对路径

- **类型**：UT
- **主体流程**：设置 `ORBION_DIR=/var/lib/orbion` 后调用各派生路径方法
- **检查项**：所有路径前缀为 `/var/lib/orbion`

### MVP-FL-1.5 orbion.json 解析新格式

- **类型**：UT
- **主体流程**：创建含 `orbion_dir` 字段的临时 orbion.json，加载 Settings
- **检查项**：`orbion_dir` 为 orbion.json 中配置的值

### MVP-FL-1.6 orbion.json 旧格式报错

- **类型**：UT
- **主体流程**：创建含 `repo_path` 字段（重构后 OrbionConfigSchema 已移除该字段）的临时 orbion.json，加载 OrbionConfigSchema
- **检查项**：校验失败，报 Extra inputs are not permitted

### MVP-FL-1.7 OrbionConfigFileParser 展开 orbion_dir

- **类型**：UT
- **主体流程**：创建含 `orbion_dir: "./custom"` 的临时 orbion.json，通过 OrbionConfigFileParser 加载
- **检查项**：data 字典中含 `orbion_dir` 键，值为 `"./custom"`

### MVP-FL-1.8 现有 test_config.py 旧断言更新

- **类型**：UT
- **主体流程**：运行现有 test_config.py 全部用例
- **检查项**：
  - test_tc1_3 不再断言 `repo_path` 和 `memory_base_path`，改为断言 `orbion_dir`
  - test_orbion_config_schema_defaults 断言 `orbion_dir` 默认值为 `"./data"`
  - 其余用例不受影响

## 步骤 2：AgentMemory 适配新路径结构

### MVP-FL-2.1 三层记忆加载

- **类型**：UT
- **主体流程**：在 tmp_path 下创建三层 memory.md（平台→项目→Agent），调用 `load_memory_chain("p1", "summary")`
- **检查项**：返回值包含三层内容，按 "平台→项目→Agent" 顺序用 `\n\n` 拼接

### MVP-FL-2.2 仅平台级记忆存在

- **类型**：UT
- **主体流程**：只创建 tmp_path/memory.md，调用 `load_memory_chain("p1", "summary")`
- **检查项**：返回值只有平台级内容

### MVP-FL-2.3 无任何记忆文件

- **类型**：UT
- **主体流程**：tmp_path 下无 memory.md 文件，调用 `load_memory_chain("p1", "summary")`
- **检查项**：返回空字符串

### MVP-FL-2.4 写入记忆文件

- **类型**：UT
- **主体流程**：对空 tmp_path 调用 `write_memory_file(settings.agent_memory_path("p1", "summary"), "新记忆")`
- **检查项**：文件 `tmp_path/projects/p1/agents/summary/memory.md` 存在且内容为 "新记忆"；中间目录自动创建

### MVP-FL-2.5 路径越界拒绝

- **类型**：UT
- **主体流程**：`orbion_dir = tmp_path`，调用 `write_memory_file(Path("/etc/passwd"), "内容")`（传入绝对路径）
- **检查项**：抛出 ValueError（resolved_path.is_relative_to(orbion_dir) 校验失败）

### MVP-FL-2.6 reset_agent_memory 清空内容

- **类型**：UT
- **主体流程**：创建 `tmp_path/projects/p1/agents/summary/memory.md`（内容 "有内容"），调用 `reset_agent_memory("p1", "summary")`
- **检查项**：文件仍存在但内容为空字符串

### MVP-FL-2.7 不再支持 correlation_id 参数

- **类型**：UT
- **主体流程**：调用 `load_memory_chain("p1", "summary", correlation_id="corr-1")`
- **检查项**：TypeError（不期望关键字参数 correlation_id）；同步更新现有 test_agent_memory.py 中使用 correlation_id 的用例

## 步骤 3：GitService 按项目隔离

### MVP-FL-3.1 仓库按项目隔离创建

- **类型**：UT
- **主体流程**：对空 tmp_path 调用 `ensure_repo("p1", "orbion")` + `ensure_repo("p2", "orbion")`
- **检查项**：两个项目各有独立 `.git/` 目录，仓库互相隔离

### MVP-FL-3.2 重复 ensure_repo 不报错

- **类型**：UT
- **主体流程**：项目 p1 仓库已存在，再次调用 `ensure_repo("p1", "orbion")`
- **检查项**：不抛异常，仓库仍可用

### MVP-FL-3.3 产出写入正确项目仓库

- **类型**：UT
- **主体流程**：p1 和 p2 仓库均已初始化，模拟 TaskOutputApproved 事件（event.project_id=p1）写入文件
- **检查项**：文件写入 p1 的 repo 下，不在 p2 目录

### MVP-FL-3.4 事件缺少 project_id 时跳过

- **类型**：UT
- **主体流程**：模拟 TaskOutputApproved 事件（event.project_id 为 None）
- **检查项**：日志输出 warning，不抛异常

### MVP-FL-3.5 get_recent_commits 按项目查询

- **类型**：UT
- **主体流程**：p1 有 1 个 commit，p2 有 2 个 commit，分别查询
- **检查项**：p1 返回 1 条，p2 返回 2 条

## 步骤 4：Agent 模板机制

### MVP-FL-4.1 创建模板

- **类型**：UT
- **主体流程**：对空 tmp_path 调用 `ensure_template("summary", declaration_dict, "默认记忆")`
- **检查项**：`agents/summary/declaration.json` 存在且为合法 JSON；`agents/summary/memory.md` 存在且内容为 "默认记忆"

### MVP-FL-4.2 重复 ensure_template 不覆盖

- **类型**：UT
- **主体流程**：模板已存在（memory.md 内容 "旧记忆"），再次调用 `ensure_template`
- **检查项**：memory.md 内容仍为 "旧记忆"，declaration.json 未变

### MVP-FL-4.3 拷贝模板到项目

- **类型**：UT
- **主体流程**：summary 模板存在，项目 p1 目录存在但无 agents 子目录，调用 `copy_to_project("summary", "p1")`
- **检查项**：`projects/p1/agents/summary/memory.md` 存在且与模板一致

### MVP-FL-4.4 项目已有 memory.md 时不覆盖

- **类型**：UT
- **主体流程**：项目 p1 的 `agents/summary/memory.md` 已存在（内容 "项目自定义记忆"），调用 `copy_to_project("summary", "p1")`
- **检查项**：memory.md 内容仍为 "项目自定义记忆"

### MVP-FL-4.5 模板不存在时不拷贝

- **类型**：UT
- **主体流程**：`agents/custom/` 模板不存在，调用 `copy_to_project("custom", "p1")`
- **检查项**：不创建文件，不抛异常

### MVP-FL-4.6 模板更新不影响已注册 Agent

- **类型**：UT
- **主体流程**：项目 p1 已从模板拷贝 `agents/summary/memory.md`（内容 "项目记忆"），更新平台模板 `agents/summary/memory.md` 内容为 "新模板记忆"，再次调用 `copy_to_project("summary", "p1")`
- **检查项**：项目 p1 的 memory.md 仍为 "项目记忆"（不覆盖已有文件）

## 步骤 5：项目创建/删除联动文件系统

### MVP-FL-5.1 项目创建 → 文件系统目录初始化

- **类型**：集成
- **主体流程**：POST /projects 创建项目
- **检查项**：`projects/<id>/` 目录存在；`projects/<id>/memory.md` 文件存在；`projects/<id>/repo/` 目录存在

### MVP-FL-5.2 项目删除 → 文件系统目录清理

- **类型**：集成
- **主体流程**：DELETE /projects/{id}（项目下有文件和 git 仓库）
- **检查项**：数据库中项目记录已删除；`projects/<id>/` 目录不存在；git 仓库随项目目录一起删除

### MVP-FL-5.3 删除项目 → 关联 Agent 记忆清理

- **类型**：集成
- **主体流程**：项目有注册的 Agent（`projects/<id>/agents/summary/memory.md` 存在），DELETE /projects/{id}
- **检查项**：Agent 记忆文件随项目目录一起删除

### MVP-FL-5.4 删除不存在的项目 → 文件系统无操作

- **类型**：集成
- **主体流程**：DELETE /projects/{id}（项目目录已在数据库删除但文件系统目录不存在）
- **检查项**：不抛异常（shutil.rmtree 的 ignore_errors=True 或目录不存在时跳过）
